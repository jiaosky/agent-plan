## 一、Java 基础

### 1. volatile 能替代 synchronized 吗？`i++` 用 volatile 修饰是否线程安全？

**追问型题目**（不背定义，直接考察理解）

- **不能替代**：volatile 只保证可见性、有序性，**不保证原子性**。`i++` 是读-改-写三步，多线程会丢更新。
- **i++ 用 volatile**：仍不安全，需配合 synchronized 或 AtomicInteger。
- **volatile 适用**：状态标志位、DCL 单例中的实例引用（仅修饰引用，不修饰自增操作）。

### 2. synchronized 是什么？有什么用？

**定义：** synchronized 是 Java 中实现互斥锁（Mutex，谐音：谬泰克斯）的机制，用于控制多线程对共享资源的访问。

**核心作用：**

- **原子性（Atomicity，谐音：阿托米西提）**：确保同一时刻只有一个线程能执行被 synchronized 保护的代码块或方法。
- **可见性（Visibility）**：线程退出 synchronized 块时，会将修改后的变量强制刷新到主内存（类似 volatile 效果）。
- **有序性（Ordering，谐音：奥德令）**：通过「锁」的语义限制指令重排序（但不如 volatile 严格）。

**使用方式：**

- 修饰实例方法（锁当前对象实例）：用于限制同一个对象实例的并发访问，典型场景如单例对象的成员方法需要保证线程安全时。
- 修饰静态方法（锁类的 Class 对象）：用于限制同一个类的所有实例或全局资源的并发访问，适合静态资源或全局逻辑处理时保证同步。
- 修饰代码块（显式指定锁对象）：用于精细控制某一段特定代码的并发访问，或多个实例、跨方法需要共同竞争同一把锁时使用，例如基于某个共享对象或自定义锁粒度进行同步。

**应用场景：**

- 对共享资源（如计数器、缓存）的复合操作（如 `i++`）。
- 保护需要原子性更新的复杂逻辑。

**synchronized 的实现原理了解吗？**

- synchronized 依赖 JVM 内部的 Monitor 对象来实现线程同步。使用的时候不用手动去 lock 和 unlock，JVM 会自动加锁和解锁。
- synchronized 加锁代码块时，JVM 会通过 `monitorenter` 、`monitorexit` 两个指令来实现同步：
  - 前者表示线程正在尝试获取 lock 对象的 Monitor；
  - 后者表示线程执行完了同步代码块，正在释放锁。

**volatile 和 synchronized 的区别？**

- **volatile** 关键字用于修饰变量，确保该变量的更新操作对所有线程是可见的，即一旦某个线程修改了 volatile 变量，其他线程会立即看到最新的值。
- **synchronized** 关键字用于修饰方法或代码块，确保同一时刻只有一个线程能够执行该方法或代码块，从而实现互斥访问。

### 3. 简述 Java 中 HashMap 的底层原理，为什么线程不安全？并发安全的是什么，为什么安全？

**原理：**

- JDK8：数组 + 链表/红黑树，`hash(key)` 计算桶位置，哈希冲突时拉链法解决。
- 链表长度 > 8 且数组长度 ≥ 64 时转为红黑树。

**线程不安全原因：**

- **死循环（JDK7）**：头插法扩容时，多线程并发导致链表成环（`get()` 时触发死循环）。原因：`transfer()` 方法中链表节点倒序迁移产生环。
- **数据覆盖（JDK8+）**：并发 `put()` 时，场景1：两线程同时触发扩容，可能导致节点丢失；场景2：计算相同桶位置时，后写入的覆盖先写入的值。
- **丢失更新（JDK8+）**：`size++` 非原子操作（读→改→写），多线程累加导致实际大小不准。

**并发安全替代：ConcurrentHashMap **

- **JDK8**：CAS（Compare And Swap，比较并交换）+ synchronized 锁细化
  - 锁机制：无冲突时 CAS 插入（如桶为空）；有冲突时 synchronized 锁住链表头节点/红黑树根节点。
  - 安全原理：原子操作（put 使用 CAS + synchronized）、可见性（volatile 修饰 table 和 Node.val）、有序性（内存屏障）。
  - 优化点：并发扩容（helpTransfer）、计数用 LongAdder 思想。
- **JDK7**：分段锁（Segment ），16 个 Segment 独立加锁，不同 Segment 的并发操作互不影响。

### 6. ThreadLocal 内存泄漏 + Java 四种引用类型

**内存泄漏原因：** ThreadLocal 的 key 是弱引用，GC 后 key = null，但 value 是强引用不会被回收；线程池中线程长期存活导致 Entry 累积。

**四种引用：** 强、软、弱、虚。ThreadLocal 用弱引用作 key，导致 key 被回收后 Entry 成 `<null, value>`，value 无法回收。

**解决：** 使用完调用 `remove()`；或改用 FastThreadLocal（Netty）。

### 7. synchronized 锁升级过程（无锁→偏向锁→轻量锁→重量锁）

- **无锁状态：** 新创建的对象还没有任何线程访问。
- **偏向锁：** 当第一个线程访问同步块时，JVM 将对象头中的标记设为偏向模式，并记录线程 ID。后续该线程进入时无需 CAS 操作，直接检查线程 ID 是否匹配。触发条件：单线程重复访问同一同步块。升级条件：发生锁竞争（第二个线程尝试获取锁）。
- **轻量级锁：** 当锁竞争时，偏向锁升级为轻量级锁。线程通过 CAS 操作自旋尝试获取锁，避免阻塞。触发条件：多个线程交替执行，无实际并行竞争。升级条件：自旋超过一定次数（默认 10 次）或自旋时第三个线程加入竞争。
- **重量级锁：** 当自旋失败，JVM 将锁升级为重量级锁（操作系统的互斥锁 Mutex），线程被阻塞并进入等待队列。特点：涉及内核态切换，开销大。释放：解锁后唤醒等待线程重新竞争。

### 8. ReentrantLock 与 synchronized 的区别？什么时候用哪个？

**区别：**


| 维度   | synchronized   | ReentrantLock              |
| ---- | -------------- | -------------------------- |
| 实现   | JVM 内置，自动加解锁   | API 显式 lock/unlock         |
| 公平锁  | 非公平            | 可指定公平/非公平                  |
| 可中断  | 不可中断           | lockInterruptibly() 可响应中断  |
| 多条件  | 单一 wait/notify | 多个 Condition               |
| 尝试获取 | 不支持            | tryLock()、tryLock(timeout) |


**选型：** 简单互斥用 synchronized；需要超时、可中断、公平锁时用 ReentrantLock。

### 2. Spring 循环依赖解决方案

**问题：** Spring 如何解决循环依赖（如 A→B→A）？

**答案：**

**核心机制：** 三级缓存 + 提前暴露对象

**解决流程：**

1. 创建 A 对象 → 存入三级缓存（singletonFactories ）
2. A 注入 B 时 → 触发 B 的创建
3. B 注入 A 时 → 从三级缓存拿到 A 的 ObjectFactory ）→ 生成 A 的代理对象 → 注入 B
4. B 完成创建 → A 注入完整 B 对象 → A 移入一级缓存

**关键限制：**

- ✅ 可解决：Setter 注入/字段注入的循环依赖
- ❌ 不可解：构造器循环依赖、Prototype 作用域循环依赖

### 3. AOP 是什么？

**定义：** AOP（Aspect-Oriented Programming， 面向切面编程）是一种编程范式，用于将横切关注点（如日志、事务）从业务逻辑中分离，通过动态代理实现代码复用。

**核心概念：**

- **切面（Aspect ）**：模块化的横切逻辑（如日志工具类）。
- **通知（Advice ）**：在连接点执行的动作（分为前置 @Before、后置 @After、环绕 @Around 等）。
- **切点（Pointcut ）**：定义哪些连接点应用通知（如 `@Pointcut("execution(* com.service.*.*(..))")`）。
- **织入（Weaving ）**：将切面代码注入到目标位置（编译期/运行期）。

**典型场景：** 事务管理（@Transactional）、权限检查（@PreAuthorize）、性能监控（统计方法执行时间）

## 三、MySQL

### 1. MVCC（Multi-Version Concurrency Control 多版本并发控制）是什么？

多版本并发控制：通过数据快照实现非锁定读，解决读写冲突。

**InnoDB实现：**

- 隐藏列：`DB_TRX_ID`（事务 ID）、`DB_ROLL_PTR`（回滚指针）。
- ReadView ：事务执行时生成，决定哪些数据版本可见（活跃事务列表）。

**作用：** 避免加锁，提高并发性能。

---

### 2. 幻读是什么？

同一事务中多次查询得到结果集行数不一致（新增/删除导致）

**解决方案：** 间隙锁/临键锁/提升隔离级别

- **间隙锁（Gap Lock ）**：适用 RR 隔离级别 + 非唯一索引查询。如 `SELECT * FROM orders WHERE amount > 1000 FOR UPDATE;` 锁 (1000, +∞) 区间，防插入。
- **临键锁（Next-Key Lock ）**：间隙锁 + 行锁（锁定左开右闭区间），如 ID=5 锁 (4,5]。
- **提升隔离级别：** `SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE`（ 串行化）完全串行化，性能差但 100% 防幻读。

**RR 不是完整解决幻读**

**避免幻读的核心机制：**

- **MVCC（快照读）**：普通 SELECT 使用 ReadView，同一事务内多次查询看到相同数据快照。限制：仅防止已存在数据的幻读（新增数据仍可能穿透）。
- **临键锁（当前读）**：`SELECT * FROM orders WHERE amount > 1000 FOR UPDATE;` 锁定 (1000, +∞) 区间。生效条件：使用 `FOR UPDATE` 或 `LOCK IN SHARE MODE`。

### 7. Change Buffer（变更缓冲）

**优化** 非唯一二级索引 的写操作性能

**运作机制：**

- **缓冲写入：** 当 INSERT/UPDATE/DELETE 修改非唯一索引时，如果索引页不在内存，操作缓存在 Change Buffer（内存结构），仅记录变更操作（非完整数据页）。
- **触发合并：** 条件1 读取该索引页时；条件2 后台线程定期合并；条件3 数据库正常关闭时。
- **持久化保证：** Change Buffer 修改会写入 Redo Log（崩溃可恢复），物理存储在系统表空间（ibdata1）。

**性能收益场景：** 写多读少的业务（如日志表）、机械硬盘环境（随机 IO 减少 60%+）

## 四、Kafka

### Kafka 如何保证消息顺序性？

**核心条件：** 单分区内的消息有序（Kafka 仅保证分区顺序）。

**实现方案：**

- **Producer（生产者）端：** 相同 key 的消息哈希到同一分区（如订单 ID）；设置 `max.in.flight.requests.per.connection=1`（关闭重试乱序）。
- **Consumer（消费者）端：** 单分区单线程消费；多分区时，在消费者内部按 key 分组处理（如内存队列）。

---

### 在手动提交 offset 的时候，如果异常导致了 offset 没有提交，后面会不会再次消费到它，为什么？

- **Offset 提交规则：** Kafka 按 offset 顺序提交（单调递增）。消费者从**上次已提交的 offset** 开始拉取消息。
- **重复消费逻辑：** 若处理完 offset 1、2、3 后，在提交前消费者崩溃，则上次提交的 offset 仍为 0。重启后会从 offset 1 重新拉取，导致 1、2、3 **重复消费**。
- **不会跳过：** Kafka 没有「跳过已处理」的机制，只认「已提交的 offset」。未提交 = 会重复投递。
- **解决方案：** 幂等消费 + 事务（如数据库唯一约束）。

### Kafka 消息积压怎么排查和解决？

**排查：** 看 Consumer Lag、消费速率、分区数是否过少、下游处理是否慢（DB、RPC）。

---

### Kafka 如何保证 Exactly Once（精确一次）？

**Producer：** 开启幂等 `enable.idempotence=true` + 事务 `transactional.id`。

**Consumer：** 消费 + 业务处理 + 提交 offset 放在同一事务（如写 DB 与提交 offset 原子化），或业务幂等 + at-least-once。

### 1. 穿透、雪崩、击崩

- **击穿：** 演唱会门票开售瞬间（热点 Key 失效），所有人涌向售票处（DB）
- **雪崩：** 双 11 零点所有优惠券同时过期（大量 Key 集体失效），导致收银系统崩溃

- **事中拦截：** 缓存空对象（注意短过期时间，如 30 秒）、恶意 IP 限流

### 2. Redis 持久化机制？RDB 和 AOF 如何选择？

- **RDB**：快照，fork 子进程写盘，恢复快，可能丢最后一次快照后的数据。
- **AOF**：追加日志，可配置 fsync 频率，数据更安全，文件大、恢复慢。

**选型：** 允许分钟级丢失用 RDB；要求高可用用 AOF（或混合）；生产常开 AOF + 定期 RDB 备份。

---

### 3. Redis 大 Key、热 Key 问题如何排查和解决？

**大 Key：** `redis-cli --bigkeys`、`MEMORY USAGE`；拆分、压缩、异步删除 UNLINK。

**热 Key：** 监控发现、本地缓存、多副本 + 读写分离、Redis Cluster 打散。

---

### 4. Redis 集群模式？主从、哨兵、Cluster 的区别？

- **主从**：读写分离，主挂需手动切换。
- **哨兵**：自动故障转移，高可用。
- **Cluster**：分片 + 高可用，16384 槽位，支持水平扩展。

## 八、性能与调优（驻场重点）

### 1. JVM 调优常用参数？如何根据业务选择 GC？

**常用参数：** -Xms/-Xmx（堆）、-XX:MetaspaceSize、-XX:+UseG1GC、-XX:MaxGCPauseMillis、-XX:+HeapDumpOnOutOfMemoryError。

**GC 选型：** 低延迟选 G1/ZGC；高吞吐选 Parallel；小堆用 Serial。结合 pause time、吞吐量、堆大小选型。

---

### 2. 如何定位和解决 Full GC 频繁？

**定位：** `jstat -gcutil` 看 GC 频率；`-XX:+PrintGCDetails` 或 GCEasy 分析；dump 看堆结构。

**常见原因：** 老年代不足、元空间不足、大对象直接进老年代、内存泄漏。对应调整堆大小、优化代码、排查泄漏。

---

### 3. 数据库连接池如何调优？连接数设多少合适？

**参数：** 最大连接数、最小空闲、超时时间、检测语句。

**经验：** 连接数 ≈ (核心数 × 2) + 有效磁盘数，或 压测得出；避免过大导致 DB 压力，过小导致等待。

---

### 4. 大批量数据导入/导出如何优化？

**导入：** 关闭索引/约束后批量 INSERT、LOAD DATA、多线程分批、调整 bulk_insert_buffer_size。

**导出：** 流式查询、分批 fetch、异步写文件、避免大 List 一次性加载。

---

### 5. 高并发写场景如何优化？有哪些手段？

**手段：** 异步化、批量合并、缓存写回、消息队列削峰、分库分表、读写分离、合理索引减少锁持有时间。

---

## 九、并发与 IO（驻场重点）

### 1. 如何设计一个高并发的秒杀系统？

**要点：** 前端限流、Redis 预减库存、MQ 异步下单、DB 扣减加乐观锁、防超卖、接口幂等、降级熔断。

---

### 2. 分布式锁如何实现？Redis 和 Zookeeper 方案对比？

**Redis：** SET key value NX EX，Lua 保证原子性；Redisson 看门狗续期。问题：主从切换可能丢锁。

**Zookeeper：** 临时顺序节点，最小节点获锁；天然高可用。对比：ZK 强一致性、Redis 性能更好。

---

### 3. 如何保证接口幂等？有哪些方案？

**方案：** 唯一约束（DB）、token 机制、状态机、分布式锁、MQ 消息去重（唯一 key）。

---

### 4. 大量请求打到 DB 如何保护？

**手段：** 限流（令牌桶、漏桶）、熔断降级、缓存、MQ 削峰、读写分离、连接池限制。

---

### 5. 异步处理有哪些方式？各适用场景？

**方式：** 线程池、CompletableFuture、MQ、@Async。场景：线程池适合轻量任务；MQ 适合解耦、持久化；@Async 适合 Spring 内异步。
