# 第 14 章：面试表达与工作沟通手册

## 1. 学习什么

本章学习如何把前面所有知识转化成面试表达和工作沟通能力。

很多人不是完全不会，而是说不出来，或者说出来像背概念。你的目标是让面试官和同事感觉：

```text
这个人不是算法专家，但确实懂 AI 应用后端怎么落地。
```

你需要掌握以下内容：

- 如何做转型自我介绍。
- 如何讲 AI 项目。
- 如何回答 RAG、SSE、Tool Calling、MCP、Agent、沙盒。
- 如何表达 Go 在 AI 后端里的价值。
- 如何回答“你不是算法工程师，为什么适合大模型公司”。
- 如何回答项目不足。
- 入职后日常沟通要会哪些关键词。
- 哪些表达会显得外行。
- 如何把传统后端经验迁移成 AI 应用后端优势。

本章学完后，你应该能说清楚：

> 我不是去装训练算法专家，而是把自己定位成 AI 应用后端工程师。我能把模型、RAG、SSE、工具调用、MCP、文件权限、沙盒和日志观测接入业务系统。

## 2. 为什么学习

面试不只是考你知道什么，也考你如何表达。

同样一句话，表达差别很大。

弱表达：

```text
我了解 RAG，就是把文档放进向量库。
```

强表达：

```text
RAG 是检索增强生成。生产里不只是向量库，还包括文件解析、切片、embedding、metadata 权限过滤、召回、rerank、Prompt 拼接、引用来源、评测和成本控制。
```

弱表达：

```text
我会 Go 的 goroutine 和 channel。
```

强表达：

```text
在 AI 后端里，我会用 goroutine 并发做向量检索和关键词检索，用 channel 承接模型 stream chunk，再通过 SSE 推给前端，同时用 context 处理用户取消和超时。
```

你要做的是把“概念词”变成“工程链路表达”。

这章就是帮你把教材变成嘴上的能力。

## 3. 知识详细内容

### 3.1 转型自我介绍

适合 30 秒版本：

> 我之前主要做传统后端，熟悉接口设计、数据库、缓存、异步任务、权限和服务稳定性。现在转向 AI 应用后端，重点补了模型调用、Prompt、RAG、SSE、Tool Calling、MCP、Agent、文件管理和沙盒安全。我不把自己定位成训练算法工程师，而是希望做大模型应用落地，把模型能力稳定、安全、可观测地接入业务系统。

适合 1 分钟版本：

> 我过去是后端工程师，主要优势在服务端工程能力，比如 API 设计、数据建模、缓存、异步任务、权限控制、日志排障和服务稳定性。最近我系统学习了 AI 应用后端，理解大模型应用的典型链路，包括用户问题进入后，如何做上下文构造、RAG 检索、Prompt 拼接、模型调用、SSE 流式输出、工具调用、权限校验和日志记录。我也准备了一个 Go AI Knowledge Assistant 项目，覆盖模型对话、SSE、文件上传、文档切片、简易 RAG 和 usage 日志。我的定位是 AI 应用后端，不是模型训练岗。

### 3.2 项目介绍模板

项目一句话：

> 我做了一个基于 Go 的 AI 知识库问答后端，支持大模型对话、SSE 流式输出、文件上传、文档切片、简易 RAG 检索、Prompt 构造、模型调用日志和基础限流。

项目链路：

```text
用户上传文件
  -> 后端保存文件
  -> 解析文本
  -> 文档切片
  -> chunk 入库

用户提问
  -> 后端检索相关 chunk
  -> 构造 RAG Prompt
  -> 调用模型
  -> SSE 流式返回
  -> 记录 usage 和日志
```

项目技术点：

- Go HTTP API。
- LLMClient 抽象。
- SSE 流式输出。
- 文件上传和切片。
- Retriever 检索接口。
- RAG Prompt。
- context 超时取消。
- usage 日志。
- 基础限流。

项目亮点：

> 我没有直接把业务代码绑死在某个模型 SDK，而是抽象了 LLMClient；RAG 检索也抽象成 Retriever，第一版用关键词检索打通链路，后续可以替换成 pgvector 或 Milvus；流式输出用 SSE，处理了 flush、done event 和 context 取消。

### 3.3 RAG 表达

被问：RAG 是什么？

回答：

> RAG 是 Retrieval Augmented Generation，检索增强生成。它的核心是先从外部知识库检索相关资料，再把资料作为上下文交给模型回答。工程上不只是 embedding 和向量库，还包括文件解析、切片、metadata 权限过滤、召回、rerank、Prompt 拼接、引用来源、评测和成本控制。

被问：RAG 难点在哪里？

回答：

> 难点主要在工程质量。文件解析不好会导致文本质量差，切片不好会影响召回，metadata 不完整会导致权限问题，top_k 和 rerank 会影响相关性，Prompt 拼接会影响模型是否基于资料回答，引用来源和评测决定能不能排查和优化。

被问：RAG 回答不准怎么排查？

回答：

> 我会先看实际召回了哪些 chunk，判断相关资料有没有被召回、是否被权限过滤、score 是否合理。如果召回不准，就查切片、embedding、top_k、关键词检索和 rerank。如果召回正确但回答错，就查 Prompt 是否要求基于资料、上下文是否过长、模型是否正确引用。

### 3.4 SSE 表达

被问：如何实现 ChatGPT 那种流式输出？

回答：

> 后端调用模型的 stream API，持续读取模型返回的 chunk。Go 里可以把 chunk 转成 channel，HTTP handler 设置 `text/event-stream`，读取 chunk 后写成 SSE event 并 flush。handler 监听 request context，用户断连或点击停止时取消上游模型请求。结束时发送 done event，并保存完整回答和 usage。

被问：SSE 和 WebSocket 怎么选？

回答：

> 如果只是大模型文本生成，主要是服务端持续推送给前端，SSE 更简单，基于 HTTP，也容易接入现有鉴权和网关。如果是实时语音、协作编辑、复杂双向控制，再考虑 WebSocket。

### 3.5 Tool Calling 表达

被问：Tool Calling 是什么？

回答：

> Tool Calling 是模型在需要外部能力时输出工具调用请求，包括工具名和参数。模型负责判断是否需要工具，后端负责校验工具名、参数、用户权限和数据权限，并执行真实工具。工具结果再回填给模型，由模型生成最终回答。

被问：工具调用怎么保证安全？

回答：

> 工具必须白名单注册，参数要按 schema 校验，再做业务校验和权限校验。查询类工具和有副作用工具要分开处理，删除、修改、发消息、执行命令这类高危操作需要二次确认或人工审批。所有工具调用都要记录 trace、user、tool、参数摘要、权限结果、状态和耗时。

### 3.6 MCP 表达

被问：MCP 是什么？

回答：

> MCP 是 Model Context Protocol，用来标准化 AI 应用和外部工具、资源、Prompt 的连接方式。Host 是 AI 应用，Client 在 Host 内连接 MCP Server，Server 暴露 Tools、Resources、Prompts。它解决的是大模型应用如何标准化接入文件、数据库、知识库、浏览器和内部系统能力。

被问：MCP 和 Tool Calling 有什么区别？

回答：

> Tool Calling 是模型使用工具的机制，模型输出工具名和参数；MCP 是工具如何被 AI 应用发现、描述和调用的协议。可以理解为 Tool Calling 解决“模型要用工具”，MCP 解决“工具怎么标准化接入”。

### 3.7 Agent 表达

被问：Agent 是什么？

回答：

> Agent 是围绕目标进行多步骤执行的系统。它会使用模型做计划和判断，调用工具获取信息或执行动作，保存任务状态，根据工具结果调整下一步，最终完成任务。后端负责工作流编排、状态管理、工具权限、失败恢复、人工确认、日志审计和成本控制。

被问：生产里 Agent 怎么设计？

回答：

> 我倾向先做 Workflow Agent，而不是完全开放式 Agent。后端控制主流程，模型负责局部判断和生成；工具调用经过白名单、参数校验和权限校验；高危操作进入 human-in-the-loop；每一步记录 trace，设置最大步骤数、最大时间和成本预算，避免无限循环。

### 3.8 沙盒表达

被问：模型执行代码或命令怎么保证安全？

回答：

> 我不会让模型直接在宿主机执行命令，而是放到沙盒里。沙盒只挂载任务需要的文件，限制工作目录、命令白名单、CPU、内存、磁盘和执行时间，默认禁网或白名单网络，不注入生产密钥。执行过程记录 trace、命令、参数摘要、退出码和 stdout/stderr 摘要。高危操作需要用户确认。

### 3.9 Go 场景表达

被问：Go 在 AI 后端里怎么用？

回答：

> Go 适合做 AI 应用后端的服务层和编排层，比如模型网关、SSE 流式服务、RAG 检索编排、Agent 状态机、MCP Server、文件异步任务、限流计费和日志观测。context 用来控制模型调用超时和取消，channel 用来转发模型 stream chunk，goroutine 用来并发检索和工具调用，interface 用来抽象模型供应商、Retriever 和 Tool。

被问：channel 用在哪里？

回答：

> 模型流式输出可以抽象成 chunk channel。模型客户端读取供应商 stream 后写入 channel，SSE handler 从 channel 读出并 flush 给前端，同时监听 context 取消。

### 3.10 项目不足怎么说

不要说：

```text
这个项目比较简单。
```

更好的说法：

> 当前版本主要目标是打通 AI 应用后端主链路，所以第一版用关键词检索模拟 RAG，没有直接接复杂向量数据库。代码上保留了 Retriever 接口，后续可以替换成 pgvector 或 Milvus，并加入 embedding、rerank、PDF 解析、多租户权限和评测集。

这样既诚实，又显示你知道下一步。

### 3.11 入职后日常沟通关键词

你要熟悉这些问法：

- 这个需求是普通模型问答，还是 RAG？
- 是否需要流式输出？
- 是否需要保留引用来源？
- 文件权限是否跟租户和部门隔离？
- 这个工具是查询类还是有副作用操作？
- 是否需要 human-in-the-loop？
- 是否要记录 token 成本？
- Prompt 是否需要版本管理？
- 是否有评测集验证回答质量？
- 这个 Agent 是否需要状态机？
- 是否要限制最大步骤数和最大工具调用次数？
- MCP Server 暴露哪些 tools？
- 沙盒是否允许访问外网？
- 删除文件后向量索引是否同步删除？

这些话在工作中很实用。

### 3.12 避坑表达

少说：

```text
RAG 就是向量库。
Agent 就是让模型自己干活。
MCP 就是调接口。
SSE 就是 WebSocket。
Prompt 能保证安全。
模型会记住历史。
工具参数模型生成了就能执行。
沙盒就是 Docker。
Go 在 AI 里没什么用。
```

换成：

```text
RAG 是检索增强生成，向量库只是其中一环。
Agent 是模型、工具、状态和工作流组成的任务系统。
MCP 是 AI 应用连接工具、资源和 Prompt 的标准协议。
SSE 适合服务端向前端推送模型流式输出。
Prompt 是约束，不是安全边界。
多轮记忆通常由后端传历史或摘要实现。
工具参数必须后端校验和鉴权。
沙盒是执行边界，不只是容器。
Go 适合 AI 应用后端的服务层和编排层。
```

## 7. 工作中典型场景

当产品说：


- 是否有文件？
