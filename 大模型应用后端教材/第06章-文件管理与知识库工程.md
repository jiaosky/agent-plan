# 第 06 章：文件管理与知识库工程

## 1. 学习什么

本章学习大模型应用后端里的文件管理与知识库工程。这里的“文件管理”不是普通的上传、下载、删除，而是围绕 AI 上下文、RAG、权限、索引和引用来源展开的一整套工程能力。

你需要掌握以下内容：

- 大模型系统里的文件为什么比传统文件上传复杂。
- 文件从上传到可被问答使用的完整生命周期。
- PDF、Word、Markdown、Excel、图片等格式的解析关注点。
- 文本抽取、清洗、切片、入库、索引之间的关系。
- 文件、文档、chunk、embedding、向量索引如何建模。
- 文件权限、租户隔离、部门权限、ACL 如何影响检索。
- 文件更新、版本管理、删除同步怎么做。
- 引用来源如何从 chunk 回到原文件、页码、段落。
- 文件解析失败、索引失败、权限错误如何排查。
- Go 后端在知识库工程里承担哪些职责。

本章学完后，你应该能说清楚：

> 大模型公司的文件管理不是简单存文件，而是把文件变成可检索、可权限控制、可引用、可更新、可删除、可审计的知识资产。

## 2. 为什么学习

传统后端里的文件管理通常是：

```text
上传文件 -> 存对象存储 -> 数据库记录路径 -> 下载或预览
```

但 AI 应用里的文件管理要复杂得多。

因为文件会成为模型的上下文来源。一个企业知识库问答系统里，用户上传的文件不仅要能下载，还要能被解析、切片、向量化、检索、引用和权限控制。

完整链路更像：

```text
上传文件
  -> 安全校验
  -> 存储原文件
  -> 创建文档记录
  -> 异步解析文本
  -> 清洗结构化
  -> 文本切片
  -> 生成 embedding
  -> 写入向量库
  -> 用户提问时召回 chunk
  -> 模型基于 chunk 回答
  -> 前端展示引用来源
```

如果文件管理没做好，RAG 一定会出问题。

常见事故包括：

- 文件上传成功，但知识库查不到。
- PDF 解析乱码。
- 表格内容丢失。
- 文档删除后仍能被问到。
- 用户看到无权限文件内容。
- 引用来源点不开。
- 文件更新了但 AI 回答还是旧内容。
- 一个租户能搜到另一个租户的资料。
- 解析任务失败但用户不知道。

所以文件管理是 RAG 的地基。面试时如果你能把文件生命周期讲清楚，会比只讲 embedding 和向量库更像真正做过工程。

## 3. 知识详细内容

### 3.1 AI 文件管理和传统文件管理的区别

| 对比项 | 传统文件管理 | AI 知识库文件管理 |
| --- | --- | --- |
| 核心目标 | 上传、下载、预览 | 让模型可检索、可引用、可回答 |
| 处理方式 | 保存原文件即可 | 解析、清洗、切片、向量化 |
| 权限关注 | 下载权限 | 检索权限、引用权限、chunk 权限 |
| 删除影响 | 删除文件记录 | 还要删除 chunk、向量、缓存 |
| 更新影响 | 替换文件 | 重新解析、重新索引、版本切换 |
| 质量问题 | 文件损坏、下载失败 | 解析错误、召回差、引用错 |
| 观测指标 | 上传成功率 | 解析成功率、索引状态、召回效果 |

传统文件系统关注“文件是否存得住”。AI 文件系统还要关注“文件是否能变成可靠上下文”。

### 3.2 文件生命周期

一个知识库文件的生命周期通常包括：

```text
上传中
  -> 上传完成
  -> 安全扫描
  -> 等待解析
  -> 解析中
  -> 解析成功
  -> 切片中
  -> embedding 中
  -> 索引完成
  -> 可检索
  -> 更新中
  -> 已删除或已归档
```

可以用状态字段表示：

```text
UPLOADING
UPLOADED
SCANNING
PARSING
PARSE_FAILED
INDEXING
INDEX_FAILED
READY
UPDATING
DELETED
ARCHIVED
```

为什么要有状态？

- 用户需要知道文件是否已经可问答。
- 后端需要支持失败重试。
- 运维需要排查卡在哪一步。
- 检索时只能检索 READY 的文档。
- 删除和更新要避免状态混乱。

### 3.3 文件上传

上传阶段要关注：

- 文件大小限制。
- 文件类型白名单。
- 文件名安全。
- MIME 类型校验。
- 重复文件处理。
- 断点续传或分片上传。
- 对象存储路径设计。
- 病毒扫描或安全扫描。
- 上传人、租户、部门记录。

文件记录建议包含：

```text
file_id
tenant_id
owner_id
department_id
original_name
storage_key
mime_type
size_bytes
sha256
status
created_at
updated_at
deleted_at
```

对象存储路径不要只用原始文件名，建议带租户和唯一 ID：

```text
tenant/{tenant_id}/files/{file_id}/original
```

这样可以避免重名、路径注入和跨租户混乱。

### 3.4 格式解析

不同文件格式的解析难点不同。

| 格式 | 难点 |
| --- | --- |
| TXT / Markdown | 相对简单，但要处理编码和标题结构 |
| PDF | 可能有扫描件、页眉页脚、分栏、表格、乱码 |
| Word | 标题层级、表格、批注、页眉页脚 |
| Excel | 多 sheet、表头、合并单元格、公式 |
| PPT | 页面顺序、备注、图文混排 |
| 图片 | OCR 质量、方向、清晰度 |
| HTML | 去噪、保留正文、链接和标题 |

解析输出不应该只是一大段纯文本，最好保留结构：

```text
page
section_title
paragraph_index
table_text
content
source_position
```

因为后续引用来源、切片、前端定位都依赖这些结构信息。

### 3.5 文本清洗

解析后的文本常常有噪声：

- 页眉页脚重复。
- 页码。
- 多余空行。
- OCR 错字。
- 表格错位。
- 目录重复。
- 水印。
- HTML 导航栏。

清洗目标不是“越干净越好”，而是保留对问答有价值的信息。

常见处理：

- 去掉重复页眉页脚。
- 合并断行。
- 保留标题层级。
- 表格转成可读文本。
- 去掉无意义符号。
- 保留页码和段落位置。

注意：不要清洗掉业务关键信息。例如合同编号、条款编号、金额、日期、产品型号都很重要。

### 3.6 文档、chunk 和 embedding 的关系

可以这样建模：

```text
File：原始文件
Document：文件解析后的文档版本
Chunk：文档切分后的文本片段
Embedding：chunk 对应的向量
Vector Index：用于相似度检索的索引
```

关系：

```text
file 1 -> document version N
document 1 -> chunks N
chunk 1 -> embedding 1
embedding -> vector index
```

推荐 chunk 字段：

```text
chunk_id
doc_id
file_id
tenant_id
content
source_page
source_section
start_offset
end_offset
embedding_status
index_status
created_at
deleted_at
```

这样可以支持：

- 从答案引用跳回原文件。
- 文件删除时清理所有 chunk。
- 文档更新时按版本重建索引。
- 检索时按租户过滤。

### 3.7 异步任务和状态机

文件解析、embedding、入向量库通常不适合同步完成。

用户上传一个 PDF 后，如果后端同步解析、切片、embedding，接口可能很慢甚至超时。

常见做法：

```text
上传接口只负责保存文件和创建任务
  -> MQ 或任务队列异步解析
  -> 解析完成后异步切片
  -> 再异步 embedding 和索引
  -> 状态更新为 READY
```

任务需要支持：

- 重试。
- 幂等。
- 失败原因记录。
- 进度查询。
- 死信队列。
- 手动重新解析。

幂等很重要。例如 embedding 任务重复执行时，不能生成重复 chunk 或重复索引。

可以用：

```text
doc_id + version + chunk_index
```

作为幂等键。

### 3.8 权限和多租户隔离

知识库文件权限比普通下载权限更敏感。

因为即使用户不能下载文件，只要能检索到 chunk，模型就可能泄露内容。

权限要贯穿：

```text
上传
  -> 文件记录
  -> chunk metadata
  -> 向量索引 metadata
  -> 检索 filter
  -> 引用查看
```

常见权限字段：

```text
tenant_id
owner_id
department_id
visibility
acl_group_ids
doc_status
```

检索时必须带权限过滤：

```text
tenant_id = 当前租户
AND doc_status = READY
AND user 可访问该文档
```

不要只在前端隐藏文件列表，也不要只靠 Prompt 告诉模型不要泄露。

### 3.9 文件更新和版本管理

文件可能会更新：

- 员工手册更新。
- 产品手册改版。
- 合同模板变更。
- FAQ 修订。

常见策略：

#### 覆盖更新

新文件替换旧文件，旧索引删除或禁用。

优点：简单。

缺点：无法回溯旧版本。

#### 版本更新

保留多个版本：

```text
doc_id: handbook
version: v1, v2, v3
active_version: v3
```

检索时只检索 active version。

优点：

- 可回滚。
- 可审计。
- 可比较版本差异。

缺点：

- 实现复杂。
- 索引管理要更严谨。

面试中可以说：

> 企业知识库更推荐版本化，至少要能保证更新时新旧索引不会同时被错误召回。

### 3.10 删除同步

文件删除不能只删文件记录。

完整删除链路：

```text
标记 file deleted
  -> 禁用 document
  -> 删除或软删 chunks
  -> 删除向量索引
  -> 清理缓存
  -> 停止未完成任务
  -> 记录审计日志
```

如果只删原文件，不删向量索引，就会出现：

```text
文件列表看不到
但 AI 还能回答文件内容
```

这在企业场景是严重问题。

删除可以用软删，但检索时必须过滤：

```text
deleted_at IS NULL
doc_status = READY
```

### 3.11 引用溯源

用户看到 AI 回答后，常常需要点击引用查看原文。

引用链路：

```text
answer citation
  -> chunk_id
  -> doc_id
  -> file_id
  -> page / section / offset
  -> 前端打开原文件定位
```

所以 chunk 必须保留：

- 文件名。
- 页码。
- 章节标题。
- 段落序号。
- 原文偏移。
- chunk_id。

Prompt 里可以给模型传：

```text
[资料1]
chunk_id: c123
source: 员工手册.pdf
page: 12
content: ...
```

后端再校验模型返回的引用编号是否存在，避免模型编造引用。

### 3.12 文件处理观测

知识库工程需要可观测。

建议记录指标：

```text
upload_success_count
parse_success_count
parse_failed_count
index_success_count
index_failed_count
avg_parse_latency
avg_embedding_latency
ready_file_count
failed_file_count
deleted_file_count
```

单文件日志建议记录：

```text
file_id
tenant_id
task_id
stage
status
error_code
error_message
duration_ms
retry_count
```

这样用户问“为什么文件不能问答”时，你能知道卡在上传、解析、embedding 还是索引。

### 3.13 Go 后端在文件知识库中的职责

Go 后端适合做：

- 文件上传 API。
- 文件 metadata 管理。
- 任务队列生产者。
- 解析任务调度。
- chunk 入库。
- embedding 调用编排。
- 向量库写入。
- 权限过滤。
- 文件状态查询。
- 删除和更新同步。
- 引用来源查询接口。

可以设计几个接口：

```go
type FileService interface {
    Upload(ctx context.Context, req UploadRequest) (*FileRecord, error)
    GetStatus(ctx context.Context, fileID string) (*FileStatus, error)
    Delete(ctx context.Context, fileID string) error
}

type Indexer interface {
    IndexDocument(ctx context.Context, docID string) error
    DeleteDocumentIndex(ctx context.Context, docID string) error
}
```

工作重点不是你亲自写 PDF 解析算法，而是把解析、切片、索引、权限、状态和任务编排串起来。

### 3.14 常见误区

误区一：文件上传成功就代表知识库可用。

不对。还要解析成功、切片成功、embedding 成功、索引成功，状态变成 READY。

误区二：删除文件只删对象存储。

不对。还要删 chunk、向量索引和缓存。

误区三：权限只控制文件下载。

不对。检索和引用同样要控制权限。

误区四：所有文件用同一种切片方式。

不对。合同、手册、FAQ、表格、代码文档应该有不同策略。

误区五：解析失败不重要。

解析失败意味着知识库无法回答，必须可见、可重试、可排查。

### 4.2 面试问答题

问题一：AI 知识库文件上传后，后端要做什么？

参考回答：

> 上传后不能只保存文件路径。后端要做文件类型和大小校验、安全扫描、保存原文件和 metadata，然后异步解析文本、清洗、切片、生成 embedding、写入向量库，并把租户、权限、来源、页码等 metadata 写到 chunk 和索引里。只有索引完成后，文件状态才能变成 READY，用户才能基于它问答。

问题二：文件删除后为什么 AI 还能回答？怎么解决？

参考回答：

> 可能是只删除了文件记录或对象存储，但 chunk、向量索引或缓存还在。正确做法是删除或禁用 file、document、chunks、vector index 和相关缓存，检索时也要过滤 deleted 状态。同时记录审计日志，并保证未完成的解析或索引任务不会把已删除文件重新写入索引。

问题三：如何做知识库多租户隔离？

参考回答：

> 文件、文档、chunk、向量索引 metadata 都要带 tenant_id，必要时还要有 department_id、owner_id、acl_group_ids。用户检索时根据身份构造权限 filter，向量检索阶段只召回当前租户且用户有权限的 chunk。不能先全库召回再交给模型判断。

问题四：文件更新如何处理？

参考回答：

> 可以使用版本化策略。新版本上传后创建新的 document version，解析、切片、embedding、索引全部成功后再切换 active_version。旧版本可以保留用于回滚或审计，但检索时只查 active version。失败时不影响旧版本可用。

### 4.3 编程题

编程题一：设计文件状态枚举。

要求包含：

- 上传中。
- 上传完成。
- 解析中。
- 解析失败。
- 索引中。
- 索引失败。
- 可检索。
- 删除。

type FileStatus string

const (

编程题三：写一个删除文件伪代码。

要求包含：

- 鉴权。
- 标记文件删除。
- 停止未完成任务。
- 删除 chunk。
- 删除向量索引。
- 清理缓存。
- 记录审计。

伪代码方向：

```text
deleteFile(user, fileID):
  file = getFile(fileID)
  checkPermission(user, file)
  markFileDeleted(fileID)
  cancelPendingTasks(fileID)

  markChunksDeleted(chunks)
  vectorStore.deleteByFileID(fileID)
  cache.delete(fileID)
  auditLog(user, "delete_file", fileID)
```

## 5. 和其他知识点的相关性

和 RAG 的关系：

- 文件管理是 RAG 的上游。
- 文件解析和切片质量直接决定 RAG 召回质量。

和 Embedding / 向量检索的关系：

- chunk 是 embedding 的输入。
- metadata 是向量检索权限过滤的基础。

和 Prompt 的关系：

- 文件 chunk 最终会进入 Prompt。
- chunk 里的来源信息会用于 Prompt 引用要求。

和 SSE 的关系：

- 文件解析、索引进度可以通过 SSE 推送。
- 大文件总结和问答也适合流式返回。

和 Tool Calling 的关系：

- Agent 可能通过工具读取文件、查询知识库、生成新文件。
- 文件权限会影响工具能访问什么。

和 MCP 的关系：

- 文件系统和知识库可以通过 MCP 暴露为 resource 或 tool。
- MCP Server 需要遵守文件权限和访问边界。

和沙盒的关系：

- 让 Agent 处理文件时，常常需要沙盒隔离。
- 沙盒里只能放任务需要的文件，不能暴露整个文件系统。

和 Go 的关系：

- Go 适合做文件 API、任务调度、状态管理、权限过滤和索引编排。
- 解析本身可以调用外部服务或独立 worker。

## 6. 演变过程与后续方向

### 6.1 从文件存储到知识资产

传统文件系统关注：

```text
文件能上传、能下载、能预览
```

AI 知识库关注：

```text
文件能被解析、能被检索、能被引用、能被权限控制
```

这意味着文件从静态资产变成了模型上下文资产。

### 6.2 从同步处理到异步管线

早期 demo 可能上传后立即解析。

生产系统会拆成异步管线：

```text
上传 -> 解析任务 -> 切片任务 -> embedding 任务 -> 索引任务
```

这样更适合大文件、高并发和失败重试。

### 6.3 从单一格式到多模态文件

最初知识库可能只支持 txt、md、pdf。

后续会扩展：

- Word。
- Excel。
- PPT。
- 图片 OCR。
- 音频转写。
- 视频字幕。
- 网页。
- 代码仓库。

多模态文件处理会越来越重要。

### 6.4 从普通权限到权限感知检索

普通文件系统只控制下载。

AI 知识库必须控制：

- 文件列表。
- 原文预览。
- chunk 检索。
- 模型上下文。
- 引用查看。
- Agent 工具读取。

后续方向是权限感知检索成为默认能力。

### 6.5 后续方向

文件和知识库工程后续会继续发展：

- 更强的文档结构解析。
- 表格理解。
- 图片和图表理解。
- 多模态 RAG。
- 文档版本差异理解。
- 知识库质量评分。
- 自动发现过期文档。
- 文档权限和组织架构联动。
- 企业级知识治理平台。

## 7. 工作中典型场景

### 场景一：用户上传文件后无法问答

排查顺序：

```text
文件是否上传成功
  -> 状态是否 READY
  -> 解析任务是否失败
  -> chunk 是否生成
  -> embedding 是否成功
  -> 向量索引是否写入
  -> 检索是否带了正确权限
```

### 场景二：PDF 解析出来全是乱码

可能原因：

- PDF 是扫描件，需要 OCR。
- 字体编码特殊。
- 分栏或表格结构复杂。
- 解析工具不适配。

处理方式：

- 对扫描件走 OCR。
- 保留页码和原文截图。
- 对复杂 PDF 使用更强解析服务。
- 允许用户查看解析结果并反馈。

### 场景三：文件删除后知识库还能搜到

排查：

- chunk 是否软删。
- 向量库是否删除。
- 检索是否过滤 deleted 状态。
- 缓存是否清理。
- 是否有未完成任务重新写入。

修复：

- 补充删除同步。
- 删除向量索引。
- 停止未完成任务。
- 加审计和告警。

### 场景四：不同部门看到彼此文件

这是权限事故。

排查：

- 文件 metadata 是否有 department_id。
- chunk metadata 是否同步写入。
- 向量检索 filter 是否带权限。
- 缓存 key 是否包含 tenant/user/permission scope。
- 引用接口是否再次鉴权。

### 场景五：文档更新后 AI 还按旧版本回答

可能原因：

- 新版本还没索引完成。
- 检索同时查了旧版本。
- active_version 没切换。
- 缓存未清理。

处理方式：

- 版本化索引。
- 新版本 READY 后再切换。
- 检索只查 active version。
- 清理相关缓存。

### 场景六：面试官问“文件管理在 AI 应用里有什么特殊”

可以这样回答：

> 传统文件管理主要是上传下载，但 AI 应用里文件会变成模型上下文。后端要负责解析、清洗、切片、embedding、索引、权限 metadata、引用来源、版本更新和删除同步。尤其是权限和删除，不能只控制原文件下载，还要确保无权限或已删除的 chunk 不会被检索给模型。
