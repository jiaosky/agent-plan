# 第 01 章：大模型应用后端全景

## 1. 学习什么

本章先建立一张全局地图：传统后端工程师进入大模型公司后，到底是在做什么，和过去写业务接口、CRUD、消息队列、定时任务有什么不同。

你需要掌握以下内容：

- 大模型应用后端的岗位定位。
- 大模型应用的典型链路。
- 传统后端和 AI 应用后端的差异。
- RAG、Agent、Tool Calling、MCP、SSE、沙盒、文件管理之间的关系。
- 一个 AI 应用后端系统通常由哪些模块组成。
- 面试时应该如何介绍自己的转型方向。
- 工作中会被问到哪些问题，如何避免显得完全外行。

本章学完后，你应该能用自己的话说清楚：

> 我不是去做模型训练的人，而是做大模型应用后端的人。我的工作是把模型能力稳定、安全、可观测、可控地接入业务系统。

## 2. 为什么学习

很多传统后端工程师转大模型方向时，会出现一个误区：一上来就学 Transformer、训练、微调、CUDA、论文。那些知识当然重要，但它们更偏算法、训练、推理底层，不是后端转 AI 应用岗位的第一优先级。

大模型公司里有很多岗位并不要求你训练模型，而是要求你把模型用好、接好、管好。

常见岗位包括：

- AI 应用后端工程师。
- Agent 平台后端工程师。
- RAG 工程师。
- 大模型产品工程师。
- AI Infra 应用层工程师。
- 模型网关工程师。
- 企业知识库后端工程师。
- 智能客服、智能办公、代码助手、数据分析 Agent 的后端工程师。

这些岗位的核心不是“你会不会推导注意力机制公式”，而是：

- 模型怎么接入业务系统。
- 用户问题怎么变成模型请求。
- 企业文档怎么变成模型可用的上下文。
- 模型如何调用工具。
- 模型输出如何流式返回。
- 模型调用失败怎么办。
- token 成本如何控制。
- 用户文件和权限如何隔离。
- 如何防止模型乱执行危险操作。
- 如何记录日志、审计、排查问题。

所以本章的意义是先帮你建立方向感。

如果方向错了，你会花大量时间学很难但短期面试用不上的东西；如果方向对了，你可以在一个月内建立起“AI 应用后端”的基本表达和项目能力。

## 3. 知识详细内容

### 3.1 大模型应用后端的核心定位

传统后端工程师通常负责：

- 设计业务接口。
- 操作数据库。
- 接入缓存、消息队列、搜索引擎。
- 做权限、事务、并发、分布式、监控。
- 保证系统稳定、可扩展、可维护。

AI 应用后端不是推翻这些能力，而是在这些能力之上增加模型相关能力。

可以这样理解：

```text
传统后端能力
  + 模型调用
  + 上下文工程
  + 检索增强
  + 工具调用
  + 流式交互
  + 成本控制
  + 安全隔离
  = AI 应用后端能力
```

你的优势是已经懂服务端工程。你要补的是模型相关的交互范式，而不是从零变成算法科学家。

### 3.2 一个大模型应用的基本链路

最简单的大模型调用链路是：

```text
用户输入 -> 后端接收请求 -> 拼 Prompt -> 调模型 API -> 返回模型回答
```

但生产系统通常不是这么简单，而是：

```text
用户输入
  -> 鉴权和限流
  -> 会话读取
  -> 意图识别
  -> 检索知识库或选择工具
  -> 构造上下文
  -> 调用模型
  -> 流式返回
  -> 工具调用处理
  -> 结果校验
  -> 日志和成本记录
  -> 会话落库
  -> 返回最终结果
```

如果有 RAG，链路会多出检索：

```text
用户问题 -> 向量检索 -> 召回文档片段 -> 拼入 Prompt -> 模型基于资料回答
```

如果有工具调用，链路会多出工具执行：

```text
用户问题 -> 模型判断需要工具 -> 后端执行工具 -> 工具结果回填模型 -> 模型总结答案
```

如果有 Agent，链路会更像任务编排：

```text
目标 -> 计划 -> 调用工具 -> 观察结果 -> 调整计划 -> 继续执行 -> 输出结果
```

这就是大模型应用后端和普通 CRUD 最大的不同：请求不是简单地查库返回，而是围绕模型进行上下文构造、工具编排和风险控制。

### 3.3 传统后端和 AI 应用后端的差异

| 对比项 | 传统后端 | AI 应用后端 |
| --- | --- | --- |
| 输入 | 参数明确的 HTTP 请求 | 自然语言、文件、图片、上下文 |
| 输出 | 结构化 JSON 为主 | 文本、JSON、工具调用、文件、流式内容 |
| 核心逻辑 | 代码规则确定 | 模型输出有不确定性 |
| 数据来源 | 数据库、缓存、第三方接口 | 数据库、文件、向量库、工具、模型上下文 |
| 交互方式 | 请求响应 | 请求响应 + SSE 流式 + 多轮会话 |
| 错误类型 | 参数错、数据库错、网络错 | 还包括幻觉、格式不稳定、上下文不够、工具误调用 |
| 成本 | 机器、数据库、带宽 | 还包括 token 成本、模型调用成本 |
| 安全 | 鉴权、越权、SQL 注入 | 还包括 Prompt Injection、工具越权、文件泄漏 |

传统后端强调确定性，AI 后端必须接受模型的不确定性，并通过工程手段把不确定性控制在业务可接受范围内。

### 3.4 AI 应用后端的核心模块

一个比较完整的 AI 应用后端，通常包含这些模块：

| 模块 | 作用 |
| --- | --- |
| API 层 | 接收聊天、文件、任务、工具调用请求 |
| 会话管理 | 保存多轮对话、上下文摘要、历史消息 |
| 模型网关 | 统一接入不同模型供应商 |
| Prompt 管理 | 管理系统提示词、模板、变量 |
| RAG 管线 | 文件解析、切片、向量化、检索、重排 |
| 工具系统 | 定义工具、校验参数、执行工具、回填结果 |
| MCP 接入 | 用标准协议连接外部工具和数据源 |
| SSE 流式输出 | 把模型增量内容实时推送给前端 |
| 沙盒系统 | 隔离执行代码、命令、文件任务 |
| 权限系统 | 控制用户、租户、文件、工具访问边界 |
| 日志观测 | 记录 trace、token、耗时、错误、成本 |
| 评测系统 | 测试回答质量、召回率、格式正确率 |

你不需要一开始全部做过，但要知道这些模块为什么存在。

### 3.5 几个核心概念的关系

可以把 AI 应用后端理解成一套“模型运行时”：

```text
LLM 是大脑
Prompt 是指令
Context 是记忆窗口
RAG 是外部知识
Tool Calling 是动手能力
MCP 是连接工具的标准接口
SSE 是实时说话方式
Sandbox 是安全执行空间
File Management 是知识和上下文来源
Observability 是排障和成本控制
```

这些概念不是孤立的。

例如，一个企业知识库问答系统至少会用到：

- 文件管理：上传和解析企业资料。
- RAG：从资料中检索相关内容。
- Prompt：把资料和问题组织给模型。
- SSE：把回答流式返回给用户。
- 权限系统：确保用户只能查自己有权限的资料。
- 日志：记录每次回答用了哪些文档和多少 token。

一个代码 Agent 系统至少会用到：

- Tool Calling：模型决定读文件、改文件、运行测试。
- MCP：把文件系统、Git、终端能力暴露为工具。
- Sandbox：限制命令执行边界。
- SSE：实时返回执行过程。
- 会话管理：保存任务状态和历史。

### 3.6 Go 后端视角

Go 在 AI 应用后端里很适合做稳定的服务层和编排层。

典型职责包括：

- 模型网关：统一封装 OpenAI、DeepSeek、通义、Claude、私有模型。
- SSE 服务：向前端流式推送模型增量内容。
- RAG 后端：管理文件上传、异步解析、检索和拼接上下文。
- Agent 编排：控制工具执行、状态流转、错误重试。
- MCP Server：把内部系统能力暴露给 Agent。
- 异步任务：处理 embedding、文件解析、批量评测。
- 成本和限流：按用户、租户、模型统计用量。

你需要重点补的 Go 能力不是“语法会不会”，而是这些场景化表达：

- `context` 用来控制模型调用超时和用户取消。
- `goroutine` 用来并发检索多个来源或并发调用工具。
- `channel` 用来转发模型流式 chunk。
- `interface` 用来抽象不同模型供应商。
- `http.Client` 用来接模型 API。
- Redis 用来做限流、会话状态、任务状态。
- MQ 用来做文件解析、embedding、异步评测。

### 3.7 常见误区

误区一：以为去大模型公司就必须会训练模型。

实际情况是，大量岗位做的是应用、平台、工程化和产品落地。你要先把 AI 应用后端做好。

误区二：只会说“我调过 ChatGPT API”。

这太浅。你要能讲模型调用后的工程问题：超时、重试、流式、成本、日志、权限、格式校验、上下文管理。

误区三：以为 RAG 就是 embedding + 向量库。

真正难点还包括文档解析、切片策略、召回质量、重排、引用来源、权限过滤、评测闭环。

误区四：以为 Agent 就是 while 循环调用模型。

生产 Agent 要考虑工具白名单、参数校验、执行权限、任务状态、人工确认、安全审计。

误区五：忽视安全。

只要模型能接触文件、工具、数据库、命令，就必须考虑 Prompt Injection、越权访问和沙盒隔离。

## 4. 考题、编程题与验收

### 4.1 概念题

1. 什么是 AI 应用后端？它和模型训练工程师有什么区别？
2. 为什么大模型应用不能简单理解为“后端调一个模型 API”？
3. RAG、Tool Calling、Agent、MCP 分别解决什么问题？
4. 为什么大模型应用经常需要 SSE 流式输出？
5. 为什么大模型应用要记录 token、耗时、trace 和成本？
6. 为什么模型输出的不确定性会影响后端设计？
7. 为什么让模型直接操作数据库或执行命令是危险的？

### 4.2 面试问答题

问题一：你是传统后端，为什么想转 AI 应用后端？

参考回答：

> 我之前做传统后端，熟悉接口、数据库、缓存、异步任务和服务稳定性。现在大模型应用落地需要大量工程能力，比如模型网关、SSE 流式输出、RAG、工具调用、文件权限、限流重试和日志观测。我不是定位训练算法岗，而是希望做 AI 应用后端，把模型能力稳定、安全、可控地接入业务系统。

问题二：如果让你设计一个企业知识库问答系统，你会从哪些模块考虑？

参考回答：

> 我会分成文件管理、解析切片、embedding、向量检索、权限过滤、Prompt 构造、模型调用、SSE 返回、引用来源、日志和评测几个模块。难点不是单纯调模型，而是文档解析质量、召回准确率、权限隔离、回答可追溯和成本控制。

问题三：大模型应用后端最容易出问题的地方有哪些？

参考回答：

> 常见问题包括模型接口超时、流式连接中断、输出格式不稳定、检索召回不准、上下文过长、token 成本失控、工具误调用、文件越权、Prompt Injection 和日志不可观测。后端需要通过超时、重试、限流、结构化输出校验、权限控制和 trace 日志降低风险。

### 4.3 编程题

编程题一：设计模型客户端接口。

要求：

- 用 Go 定义一个 `LLMClient` 接口。
- 支持普通聊天和流式聊天。
- 支持 `context.Context`。
- 能表达模型名称、消息列表、温度、最大输出 token。

参考方向：

```go
type Message struct {
    Role    string
    Content string
}

type ChatRequest struct {
    Model       string
    Messages    []Message
    Temperature float64
    MaxTokens   int
}

type ChatResponse struct {
    Content      string
    InputTokens  int
    OutputTokens int
}

type ChatChunk struct {
    Delta        string
    FinishReason string
}

type LLMClient interface {
    Chat(ctx context.Context, req ChatRequest) (*ChatResponse, error)
    Stream(ctx context.Context, req ChatRequest) (<-chan ChatChunk, error)
}
```

验收标准：

- 能解释为什么接口里要有 `context.Context`。
- 能解释为什么要抽象 `LLMClient`。
- 能解释普通返回和流式返回的区别。

编程题二：画出聊天请求的处理伪代码。

要求包含：

- 鉴权。
- 限流。
- 读取会话。
- 构造 Prompt。
- 调用模型。
- 记录日志。
- 返回结果。

伪代码方向：

```text
handleChat(req):
  user = auth(req)
  rateLimit(user)
  session = loadSession(req.sessionID)
  messages = buildMessages(session, req.question)
  resp = llm.chat(ctx, messages)
  saveMessage(session, req.question, resp.content)
  recordUsage(user, resp.tokens, resp.latency)
  return resp.content
```

### 4.4 系统设计题

题目：设计一个面向企业内部员工的 AI 助手后端。

要求回答：

- 如何接入多个模型？
- 如何支持流式输出？
- 如何接入企业知识库？
- 如何保证不同部门文档权限隔离？
- 如何记录模型调用成本？
- 如果模型返回格式不稳定，如何处理？
- 如果用户要求 AI 执行危险操作，如何拦截？

验收标准：

- 回答中必须出现模型网关、RAG、权限过滤、SSE、日志、限流、安全审计。
- 不能只说“调模型 API”。

## 6. 演变过程与后续方向

### 6.1 从传统后端到 AI 应用后端

传统后端的核心链路是：

```text
请求 -> 参数校验 -> 业务逻辑 -> 数据库 -> 返回 JSON
```

AI 应用后端的链路逐步演变成：

```text
请求 -> 上下文构造 -> 模型推理 -> 工具或检索增强 -> 流式返回 -> 质量和安全控制
```

这不是替代传统后端，而是把传统后端的工程能力扩展到模型交互领域。

### 6.2 从 Chatbot 到 RAG

早期很多应用只是聊天机器人：

```text
用户问题 -> 模型回答
```

问题是模型不知道企业内部知识，容易胡说。

于是出现 RAG：

```text
用户问题 -> 检索企业知识 -> 模型基于资料回答
```

RAG 让模型能接入企业文档、规章制度、客服知识库、代码仓库、产品手册。

### 6.3 从 RAG 到 Agent

RAG 主要解决“知道什么”的问题。

Agent 进一步解决“能做什么”的问题。

例如：

- 查订单。
- 改工单状态。
- 调用报表接口。
- 生成 SQL。
- 修改代码。
- 运行测试。
- 创建文档。

这就要求后端提供工具系统、权限控制、执行审计和沙盒环境。

### 6.4 从私有接口到 MCP

早期每个 Agent 框架都有自己的工具接入方式。这样会导致工具生态割裂。

MCP 的方向是把工具、资源、Prompt 用更标准的方式暴露给模型应用，让 AI 应用能更容易连接文件系统、数据库、浏览器、内部平台和开发工具。

后续方向大概率是：

- 更多内部系统会被 MCP 化。
- Agent 平台会内置工具市场。
- 权限、审计、沙盒会成为标配。
- AI 应用后端会越来越像“模型操作系统”的服务层。

### 6.5 后续你应该关注的方向

短期优先：

- 模型调用工程化。
- RAG。
- SSE。
- Tool Calling。
- MCP。

长期再看：

- 推理加速。
- 多模态。
- 端侧模型。
- AI Infra 底层。

## 7. 工作中典型场景

### 场景一：产品要做一个 AI 知识库问答

你需要追问：

- 文档来源是什么？
- 支持哪些格式？
- 是否需要权限隔离？
- 是否要引用来源？
- 更新文档后索引如何刷新？
- 回答错误如何评测和纠正？

工程方案通常包括：

- 文件上传。
- 文档解析。
- 文本切片。
- embedding。
- 向量检索。
- rerank。
- Prompt 拼接。
- 模型调用。
- SSE 返回。
- 引用来源。
- 调用日志。

### 场景二：前端说模型回答太慢

你需要判断：

- 是模型首 token 慢，还是整体生成慢？
- 是否用了流式输出？
- Prompt 是否太长？
- 检索是否太慢？
- 模型供应商是否超时？
- 是否需要缓存或降级模型？

常见处理：

- 使用 SSE。
- 缩短上下文。
- 并发检索。
- 设置超时。
- 记录耗时分段。
- 按场景选择更快模型。

### 场景三：模型回答引用了用户无权限的文件

这是严重问题。

你需要检查：

- 检索前是否做权限过滤。
- 向量库 metadata 是否带用户、部门、租户信息。
- 文件删除后索引是否同步删除。
- 日志是否能追踪召回了哪些 chunk。

正确原则：

> 权限过滤必须发生在检索链路里，不能只靠 Prompt 告诉模型“不要回答无权限内容”。

### 场景四：模型想调用危险工具

例如用户说：

```text
帮我删除所有测试库数据。
```

你不能直接让模型执行。

需要：

- 工具白名单。
- 参数校验。
- 用户权限判断。
- 高危操作二次确认。
- 审计日志。
- 沙盒隔离。

### 场景五：老板问 token 成本为什么暴涨

你需要能排查：

- 哪个用户或租户调用最多。
- 哪个模型成本最高。
- Prompt 是否过长。
- 是否重复发送历史消息。
- 是否 RAG 召回过多 chunk。
- 是否缺少缓存。
- 是否有异常重试。

需要日志字段：

- trace_id。
- user_id。
- tenant_id。
- model。
- input_tokens。
- output_tokens。
- latency_ms。
- status。
- error_code。
- request_type。
