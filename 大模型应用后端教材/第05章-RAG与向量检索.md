# 第 05 章：RAG 与向量检索

## 1. 学习什么

本章学习大模型应用后端里最常见、最能体现工程能力的方向：RAG 与向量检索。

RAG 是 Retrieval Augmented Generation，检索增强生成。它的核心思想是：不要只依赖模型自己“记得什么”，而是在回答前先从外部知识库检索相关资料，再把资料作为上下文提供给模型，让模型基于资料回答。

你需要掌握以下内容：

- RAG 是什么，解决什么问题。
- 为什么企业知识库、客服问答、制度问答、代码问答都需要 RAG。
- RAG 的离线链路和在线链路。
- 文档解析、切片、Embedding、向量库、召回、rerank、Prompt 拼接之间的关系。
- 向量检索和关键词检索的区别。
- chunking 切片策略为什么重要。
- top_k、相似度、metadata filter、rerank 是什么。
- 引用来源和权限过滤怎么做。
- RAG 的常见失败原因和排查方法。
- Go 后端在 RAG 系统里负责什么。

本章学完后，你应该能说清楚：

> RAG 不是简单地“把文档丢进向量库”。一个可用的 RAG 系统要处理文件解析、切片、Embedding、索引、权限过滤、召回、重排、上下文拼接、引用来源、评测和成本控制。

## 2. 为什么学习

大模型本身有几个天然限制：

- 不知道企业内部最新资料。
- 不知道你的私有数据库和文档。
- 可能编造不存在的信息。
- 训练知识有时间边界。
- 对具体制度、合同、项目文档不一定准确。
- 上下文窗口有限，不能每次塞入所有资料。

例如用户问：

```text
我们公司试用期员工有没有年假？
```

如果直接问模型，模型可能根据通用劳动常识回答，但它并不知道你公司的内部制度。

RAG 的思路是：

```text
先找到公司制度里相关段落
  -> 再让模型基于这些段落回答
  -> 回答时附上引用来源
```

这对企业应用非常关键。

典型场景：

- 企业知识库问答。
- 智能客服。
- 内部规章制度问答。
- 产品手册问答。
- 合同条款问答。
- 代码仓库问答。
- 法务、财务、人事资料检索。
- 售前方案生成。
- 运维故障知识库。

面试中，RAG 经常是 AI 应用后端绕不开的问题。面试官不只会问“RAG 是什么”，还会继续追：

- 文档怎么切片？
- 为什么要用 Embedding？
- 向量库里存什么？
- 怎么做权限过滤？
- 检索不准怎么办？
- 为什么回答没有引用？
- 如何评估 RAG 效果？
- 文件更新或删除后索引怎么办？
- top_k 取多少？
- rerank 有什么用？

所以这章是你从“知道概念”到“能讲工程”的关键。

## 3. 知识详细内容

### 3.1 RAG 的核心思想

RAG 可以拆成两个动作：

```text
Retrieval：先检索相关资料
Generation：再让模型基于资料生成回答
```

普通问答：

```text
用户问题 -> 模型 -> 回答
```

RAG 问答：

```text
用户问题 -> 检索知识库 -> 拿到相关片段 -> 拼进 Prompt -> 模型 -> 回答 + 引用
```

它解决的是：

- 模型不知道私有知识。
- 模型容易幻觉。
- 企业回答需要可追溯。
- 文档太多不能全部放进上下文。

一句面试表达：

> RAG 的本质是用检索系统给模型补充外部上下文，让模型从“凭记忆回答”变成“基于召回资料回答”。

### 3.2 RAG 的两条链路：离线和在线

RAG 系统通常分为离线入库链路和在线问答链路。

离线链路：

```text
文件上传
  -> 文件解析
  -> 文本清洗
  -> 文本切片
  -> 生成 Embedding
  -> 写入向量库
  -> 写入 metadata
```

在线链路：

```text
用户提问
  -> 问题改写或向量化
  -> 向量检索
  -> metadata 权限过滤
  -> top_k 召回
  -> rerank 重排
  -> 拼接 Prompt
  -> 调用模型
  -> 返回答案和引用
```

这两条链路要分开理解。

很多新手只关注在线提问，却忽视离线入库。实际上 RAG 效果差，经常不是模型差，而是文件解析、切片、索引、metadata 做得差。

### 3.3 Embedding 是什么

Embedding 是把文本转换成向量。

例如：

```text
"试用期员工有没有年假？" -> [0.12, -0.03, 0.77, ...]
```

向量的意义是让机器能计算文本之间的语义相似度。

例如：

```text
问题：试用期员工有没有年假？
文档片段A：试用期员工休假规定...
文档片段B：办公用品采购流程...
```

即使问题和片段 A 不完全使用相同词，Embedding 也可能判断它们语义接近。

Embedding 常用于：

- 文本相似度搜索。
- 知识库召回。
- 聚类。
- 去重。
- 推荐。

面试表达：

> Embedding 的作用是把文本映射到向量空间，使语义相近的文本在向量空间中距离更近，从而可以用向量检索召回相关文档片段。

### 3.4 向量数据库存什么

向量库不是只存一个向量。

通常存：

```text
chunk_id：片段 ID
doc_id：文档 ID
tenant_id：租户 ID
owner_id：拥有者
department_id：部门
content：文本片段
embedding：向量
source：来源文件
page：页码
section：章节
created_at：创建时间
version：文档版本
metadata：其他过滤字段
```

为什么 metadata 很重要？

因为企业 RAG 不能只查“最相似”，还要查“用户有权限的最相似”。

正确链路应该是：

```text
先按租户、用户、部门、文档状态过滤
  -> 再在可访问范围内做向量相似度检索
```

或者至少在检索过程中带 metadata filter，不能检索后才把无权限内容交给模型。

### 3.5 文本切片 chunking

为什么要切片？

因为文档通常很长，不能整篇直接向量化或塞给模型。

切片的目的：

- 控制每个片段长度。
- 保持语义完整。
- 提高召回精度。
- 降低上下文成本。
- 方便引用来源。

常见切片方式：

| 切片方式 | 说明 | 适用场景 |
| --- | --- | --- |
| 固定长度切片 | 按字符或 token 长度切 | 简单文本文档 |
| 按段落切片 | 按自然段切 | 规章制度、文章 |
| 按标题层级切片 | 按章节结构切 | 手册、规范文档 |
| 滑动窗口切片 | 相邻片段有 overlap | 避免上下文断裂 |
| 语义切片 | 按语义边界切 | 高质量知识库 |

切片过大：

- 召回不精准。
- Prompt 成本高。
- 噪声多。

切片过小：

- 语义不完整。
- 模型拿到片段后无法回答。
- 引用碎片化。

常见经验：

```text
保留标题
保留段落边界
适当 overlap
保留来源和页码
按业务文档类型调整策略
```

### 3.6 向量检索和关键词检索

关键词检索：

```text
根据字面词匹配，例如 BM25、倒排索引
```

向量检索：

```text
根据语义相似度匹配
```

对比：

| 方式 | 优点 | 缺点 |
| --- | --- | --- |
| 关键词检索 | 精确词匹配强，适合专有名词 | 同义表达召回弱 |
| 向量检索 | 语义召回强，适合自然语言问题 | 可能召回语义相近但不精确内容 |
| 混合检索 | 结合两者优势 | 实现和调参更复杂 |

例如用户问：

```text
试用期有没有假期？
```

文档写的是：

```text
新员工试用阶段休假安排...
```

关键词检索可能不稳定，向量检索可能更容易召回。

但如果用户问：

```text
CRM-2024-Refund-Rule 这个规则是什么？
```

关键词检索可能更靠谱。

成熟系统常用混合检索：

```text
向量召回 + 关键词召回 + 合并去重 + rerank
```

### 3.7 top_k、相似度和阈值

top_k 是取最相似的前 K 个片段。

例如：

```text
top_k = 5
```

表示取相似度最高的 5 个 chunk。

top_k 太小：

- 可能漏掉答案。

top_k 太大：

- Prompt 变长。
- 成本增加。
- 噪声增加。
- 模型容易被无关片段干扰。

相似度阈值用于过滤低相关片段。

例如：

```text
只保留 score > 0.75 的片段
```

但阈值不能死记。不同 embedding 模型、向量库、距离算法，分数分布可能不同。

工程上要通过评测集调：

- top_k。
- score threshold。
- chunk size。
- overlap。
- rerank 开关。

### 3.8 rerank 重排

向量召回通常是第一阶段，速度快，但不一定最准确。

rerank 是第二阶段：

```text
先召回较多候选 chunk
  -> 再用重排模型判断问题和每个 chunk 的相关性
  -> 重新排序
  -> 取前几个给大模型
```

例如：

```text
向量召回 top 20
rerank 后取 top 5
```

rerank 的价值：

- 提高最终上下文相关性。
- 减少噪声片段。
- 提高回答质量。

代价：

- 增加延迟。
- 增加成本。
- 多一个服务依赖。

面试表达：

> 向量检索适合快速粗召回，rerank 适合在候选结果里做更精细的相关性排序。企业知识库如果回答质量要求高，通常会考虑 hybrid search + rerank。

### 3.9 引用来源 citation

企业 RAG 很重要的一点是可追溯。

用户不只想要答案，还想知道：

- 答案来自哪个文件？
- 哪一页？
- 哪个段落？
- 是否有原文依据？

所以 chunk 需要保留来源信息：

```text
doc_id
file_name
page
section_title
chunk_id
content
```

Prompt 中也要明确要求模型返回引用：

```text
回答时请标注引用资料编号，如 [资料1] [资料3]。
如果资料不足，请回答“资料不足，无法确认”。
```

后端最好不要完全相信模型生成的引用。更稳的做法是：

- 后端知道本次传给模型的 chunk 列表。
- 模型只能从这些编号中选择引用。
- 服务端校验引用编号是否合法。
- 前端点击引用时，根据 chunk_id 打开原文位置。

### 3.10 权限过滤

RAG 权限是高危点。

错误做法：

```text
先从全库检索相关文档
  -> 把结果给模型
  -> Prompt 告诉模型不要泄露无权限内容
```

这是不安全的。

正确原则：

> 模型不应该看到用户无权限的内容。

权限过滤应发生在检索前或检索过程中：

```text
用户身份
  -> 计算可访问文档范围
  -> 向量检索时带 tenant_id / department_id / acl filter
  -> 只召回有权限的 chunk
  -> 再拼给模型
```

metadata 里通常要有：

```text
tenant_id
department_id
owner_id
visibility
acl_group_ids
doc_status
```

工作中如果发生“模型回答了无权限文档内容”，这通常是严重事故。

### 3.11 RAG 的常见失败原因

RAG 效果不好，不要第一反应怪模型。

常见问题：

| 问题 | 可能原因 |
| --- | --- |
| 回答不知道 | 没召回相关 chunk，或资料确实没有 |
| 回答胡说 | Prompt 没约束基于资料，或资料噪声太多 |
| 引用错误 | chunk 编号混乱，服务端没校验 |
| 检索不准 | 切片差、embedding 不合适、top_k 不合理 |
| 答案缺细节 | chunk 太小，语义不完整 |
| 成本太高 | 召回片段太多，上下文过长 |
| 越权回答 | metadata 过滤缺失或错误 |
| 更新不生效 | 文档更新后索引没刷新 |
| 删除后还能查到 | 向量索引没有同步删除 |

排查顺序：

```text
用户问题是什么
  -> 实际召回了哪些 chunk
  -> chunk 是否有权限
  -> chunk 是否相关
  -> Prompt 怎么拼的
  -> 模型是否基于资料回答
  -> 引用是否来自传入 chunk
```

### 3.12 RAG 评测

不要只靠主观感觉判断 RAG 好不好。

需要准备一批评测问题：

```text
问题
标准答案
应召回文档
应召回 chunk
是否需要拒答
权限场景
```

常见指标：

- 召回率：相关资料有没有被找出来。
- 精确率：召回结果里无关内容多不多。
- answer correctness：答案是否正确。
- citation accuracy：引用是否正确。
- refusal accuracy：资料不足时是否拒答。
- latency：检索和生成耗时。
- cost：token 和模型成本。

后端工程师不一定要搭完整评测平台，但要知道：

> RAG 优化必须有评测集，否则只是凭感觉调参数。

### 3.13 Go 后端在 RAG 中做什么

Go 后端通常负责：

- 文件上传接口。
- 解析任务入队。
- 调用解析服务。
- 文本切片。
- 调 embedding 服务。
- 写向量库和 metadata。
- 提供检索接口。
- 做权限过滤。
- 拼接 Prompt。
- 调模型。
- SSE 返回。
- 记录日志和引用。
- 文档更新和删除后的索引同步。

可以抽象接口：

```go
type DocumentChunk struct {
    ID       string
    DocID    string
    Source   string
    Content  string
    Metadata map[string]string
}

type Retriever interface {
    Search(ctx context.Context, req SearchRequest) ([]DocumentChunk, error)
}

type SearchRequest struct {
    Query    string
    TenantID string
    UserID   string
    TopK     int
    Filters  map[string]string
}
```

注意：接口里有 `TenantID`、`UserID`、`Filters`，说明检索不是只按相似度，还要带权限和业务过滤。

### 3.14 常见误区

误区一：RAG 等于向量数据库。

向量库只是 RAG 的一个组件。RAG 还包括文档解析、切片、Embedding、召回、重排、Prompt、引用、权限和评测。

误区二：Embedding 模型随便选。

不同语言、领域、文本长度对 Embedding 效果有影响。中文、代码、法律、医疗、客服知识库可能需要不同评估。

误区三：top_k 越大越好。

top_k 太大可能引入噪声，让模型更容易跑偏，也增加 token 成本。

误区四：权限靠 Prompt 控制。

权限必须在后端检索链路做，不能把无权限内容交给模型。

误区五：文件删除只删数据库记录。

还要删除对应 chunk、向量索引和缓存，否则仍可能被检索到。

### 4.2 面试问答题

问题一：你怎么理解 RAG？

参考回答：

> RAG 是检索增强生成。它不是让模型凭训练知识回答，而是在回答前先从外部知识库检索相关文档片段，把这些片段作为上下文交给模型生成答案。工程上要处理文件解析、切片、Embedding、向量检索、权限过滤、rerank、Prompt 拼接、引用来源和效果评测。

问题二：如果让你设计企业知识库问答，你会怎么做？

参考回答：

> 我会分成离线入库和在线问答两条链路。离线链路处理文件上传、解析、清洗、切片、Embedding、写入向量库和 metadata。在线链路对用户问题向量化，带租户和权限过滤检索 top_k chunk，必要时 rerank，再把片段带来源编号拼进 Prompt，调用模型生成答案并返回引用。日志里记录 trace、chunk_id、token、耗时和用户信息。

问题三：RAG 回答不准，你怎么排查？

参考回答：

> 我会先看实际召回了哪些 chunk，判断是否有相关资料、是否被权限过滤、相似度分数如何。如果没召回，检查切片、embedding、top_k、关键词召回和 rerank。如果召回正确但回答错，再检查 Prompt 是否要求基于资料回答、上下文是否过长、引用是否正确、模型输出是否被校验。

问题四：如何做 RAG 权限隔离？

参考回答：

> 文档和 chunk 入库时要写入 tenant_id、department_id、owner_id、acl_group 等 metadata。用户提问时先计算可访问范围，向量检索时带 metadata filter，只召回用户有权限的 chunk。不能先从全库召回再靠 Prompt 要求模型不泄露，因为模型不应该看到无权限内容。

### 4.3 编程题

编程题一：设计 RAG 检索接口。

要求：

- 输入包含 query、tenant_id、user_id、top_k、filters。
- 输出包含 chunk_id、doc_id、source、content、score。
- 支持 context。

参考方向：

```go
type SearchRequest struct {
    Query    string
    TenantID string
    UserID   string
    TopK     int
    Filters  map[string]string
}

type SearchResult struct {
    ChunkID string
    DocID   string
    Source  string
    Content string
    Score   float64
}

type Retriever interface {
    Search(ctx context.Context, req SearchRequest) ([]SearchResult, error)
}
```

验收标准：

- 能解释为什么要有 tenant_id 和 user_id。
- 能解释为什么结果里要有 source。
- 能解释 score 的作用。

编程题二：写一个 RAG 问答伪代码。

要求包含：

- 鉴权。
- 检索。
- 权限过滤。
- Prompt 拼接。
- 模型调用。
- 引用返回。
- 日志记录。

伪代码方向：

```text
answer(question, user):
  auth(user)
  filters = buildPermissionFilters(user)
  chunks = retriever.search(question, filters, topK=8)
  reranked = rerank(question, chunks)
  selected = takeTop(reranked, 5)
  messages = buildRAGPrompt(question, selected)
  resp = llm.chat(messages)
  citations = validateCitations(resp.citations, selected)
  recordTrace(user, question, selected, resp.usage)
  return answer + citations
```

编程题三：设计 chunk 表结构。

要求字段：

- chunk_id。
- doc_id。
- tenant_id。
- content。
- embedding_id。
- source。
- page。
- section。
- version。
- created_at。
- deleted_at。

验收标准：

- 能按文档删除 chunk。
- 能按租户过滤。
- 能定位引用来源。
- 能处理文档更新版本。

## 5. 和其他知识点的相关性

和 LLM 调用的关系：

- RAG 最终仍然要调用 LLM。
- 检索结果会成为模型调用的 messages 上下文。

和 Prompt 的关系：

- RAG 检索出的 chunk 必须通过 Prompt 清晰组织。
- Prompt 要告诉模型只能基于资料回答，并返回引用。

和 SSE 的关系：

- RAG 的检索阶段可能不流式，生成阶段通常流式。
- 更好的体验是把“正在检索”“正在生成”作为事件推给前端。

和文件管理的关系：

- RAG 的数据源来自文件。
- 第 06 章会继续展开文件上传、解析、生命周期、删除同步和权限。

和 Tool Calling 的关系：

- RAG 是“查知识”，Tool Calling 是“做动作”。
- 很多 Agent 会先 RAG 查资料，再调用工具执行操作。

和 MCP 的关系：

- MCP 可以把知识库检索能力暴露为 tool 或 resource。
- Agent 可以通过 MCP 调用企业搜索或文档系统。

和 Agent 的关系：

- Agent 经常需要检索外部资料辅助规划和执行。
- RAG 是 Agent 的重要外部记忆和知识来源。

和 Go 的关系：

- Go 适合做 RAG 服务的 API 层、任务调度、权限过滤、检索编排和模型调用。
- embedding 计算可以调外部服务，向量库可以接 pgvector、Milvus、Qdrant 等。

## 6. 演变过程与后续方向

### 6.1 从纯模型问答到 RAG

早期问答系统可能直接问模型：

```text
用户问题 -> 模型回答
```

问题是模型不知道企业私有知识，且容易编造。

RAG 加入外部知识：

```text
用户问题 -> 检索资料 -> 模型基于资料回答
```

这让大模型更适合企业落地。

### 6.2 从简单向量检索到混合检索

最初很多系统只做：

```text
query embedding -> vector top_k
```

后来发现：

- 专有名词需要关键词匹配。
- 编号、代码、合同条款更适合精确检索。
- 语义相似不等于答案相关。

所以逐步发展为：

```text
向量检索 + 关键词检索 + metadata filter + rerank
```

### 6.3 从粗糙切片到结构化知识处理

早期切片可能只是固定长度切。

成熟系统会考虑：

- 标题层级。
- 表格结构。
- 页码。
- 段落语义。
- 图片 OCR。
- 代码块。
- FAQ 问答对。
- 文档版本。

RAG 效果越来越依赖文档处理质量。

### 6.4 从“能回答”到“可评测、可运营”

demo 只看能不能回答。

生产系统还要看：

- 答案是否准确。
- 引用是否正确。
- 哪些问题答不上来。
- 哪些文档被频繁引用。
- 哪些召回结果低质量。
- 哪些用户成本高。
- 哪些文档需要更新。

RAG 会从技术功能演变成可运营的知识系统。

### 6.5 后续方向

RAG 后续会继续发展：

- Hybrid Search 混合检索。
- Graph RAG，结合知识图谱。
- Agentic RAG，让 Agent 决定如何检索和多轮查询。
- 多模态 RAG，支持图片、表格、音视频。
- Query Rewrite，自动改写用户问题。
- Self-RAG，模型自我判断是否需要检索。
- RAG 评测自动化。
- 权限感知检索。
- 企业知识治理和持续更新。

## 7. 工作中典型场景

### 场景一：产品要做企业知识库问答

你需要追问：

- 文档来源有哪些？
- 支持哪些格式？
- 是否多租户？
- 是否有部门权限？
- 是否需要引用来源？
- 文档更新频率多高？
- 答不上来时怎么处理？
- 是否要统计用户问题和命中率？

工程方案：

- 文件上传。
- 解析和切片。
- embedding。
- 向量库和 metadata。
- 权限过滤检索。
- rerank。
- Prompt 拼接。
- SSE 输出。
- 引用来源。
- 日志和评测。

### 场景二：用户说“知识库里有，但 AI 答不上来”

排查步骤：

```text
确认文档是否入库
  -> 确认 chunk 是否生成
  -> 确认 embedding 是否成功
  -> 用用户问题看召回结果
  -> 检查权限过滤是否误杀
  -> 检查 top_k 和相似度
  -> 检查 Prompt 是否包含相关 chunk
```

### 场景三：模型回答了无权限内容

这是严重问题。

排查：

- 向量检索是否带 tenant_id。
- metadata 是否正确。
- 用户权限计算是否正确。
- 是否从缓存返回了其他用户结果。
- Prompt 中是否包含无权限 chunk。

处理：

- 修复权限过滤。
- 清理错误缓存。
- 补充审计日志。
- 回放相关 trace。

### 场景四：回答太啰嗦且成本高

可能原因：

- top_k 太大。
- chunk 太长。
- 历史消息太多。
- Prompt 重复。
- 没有限制 max_tokens。

处理：

- 降低 top_k。
- 增加 rerank。
- 优化 chunk。
- 压缩 Prompt。
- 设置输出长度。

### 场景五：文档删除后仍然能被问到

可能原因：

- 只删了文件记录。
- chunk 表没删。
- 向量库没删。
- 缓存没清。
- 文档版本状态没过滤。

正确处理：

```text
删除文件
  -> 标记 doc deleted
  -> 删除或禁用 chunks
  -> 删除向量索引
  -> 清理缓存
  -> 记录审计
```

### 场景六：面试官问“RAG 难点在哪里”

可以这样回答：

> RAG 的难点不在调用 embedding 或向量库，而在文档解析质量、切片策略、metadata 权限过滤、召回准确率、rerank、上下文拼接、引用来源、更新删除同步、评测闭环和成本控制。生产系统要能看到每次问题召回了哪些 chunk，否则很难排查回答错误。
