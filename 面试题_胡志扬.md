## 一、Golang 基础能力（问题-答案-追问-答案）

### 1. slice 与 map 基础

**面试题**：Go 里 slice 的 len 和 cap 有什么区别？对 slice 做 append 时，什么情况下会触发扩容？扩容规则大概是什么？

**参考答案要点：**
- **len**：当前元素个数；**cap**：底层数组容量
- **扩容**：append 时超过 cap 会扩容；一般 2 倍（超过 1024 后约 1.25 倍）

**追问入口：**
- 若答出扩容 → "扩容后原来的 slice 和新的 slice 指向的底层数组一样吗？"
- 若答得流畅 → "nil slice 和空 slice（`s := []int{}`）有什么区别？"

**追问参考答案：**
- 扩容会分配新底层数组，返回的新 slice 指向新数组，原 slice 仍指向旧数组
- nil slice 是 `var s []int`，len=0、cap=0、未分配底层数组；空 slice 已分配底层数组但无元素；json 序列化时 nil 为 null，空 slice 为 []

---

**面试题**：map 的 key 有什么限制？对 nil map 做读操作会 panic 吗？做写操作呢？

**参考答案要点：**
- **key**：必须可比较类型（不含 slice、map、func）
- **nil map**：读返回零值不 panic；写会 panic，需先 make

**追问入口：**
- 若答出 → "为什么 slice 不能作为 map 的 key？"
- 若答得流畅 → "for range map 时遍历顺序是固定的吗？"

**追问参考答案：**
- slice 不可比较，无法作为 key
- map 遍历顺序随机，不要依赖顺序

---

### 2. 值类型与引用类型、传参

**面试题**：Go 里函数传参时，slice 是值传递还是引用传递？为什么在函数里 append slice 可能不生效？

**参考答案要点：**
- **传参**：slice 是值传递，但传的是「 slice 头」（指针+len+cap），底层数组共享
- **append 不生效**：append 可能触发扩容，新元素写入新数组，原 slice 的 len/cap 未变，调用方看不到新元素；需返回新 slice 或传指针

**追问入口：**
- 若答得含糊 → "在函数里修改 slice 的已有元素（如 s[0]=1），调用方能看到吗？"
- 若答得流畅 → "map 传参后，在函数里修改会影响原 map 吗？"

**追问参考答案：**
- 修改已有元素会影响，因为底层数组共享
- map 传的也是「头」，修改会影响原 map

---

### 3. defer 与 recover

**面试题**：`defer` 的执行顺序是什么？多个 defer 时，先 defer 的和后 defer 的谁先执行？`recover` 必须在什么地方调用才能捕获 panic？

**参考答案要点：**
- **顺序**：后 defer 的先执行（LIFO，栈）
- **recover**：必须在 defer 的**直接调用**里才生效；且不能跨 goroutine 捕获

**追问入口：**
- 若答出 defer → "defer 后面的函数，参数是什么时候求值的？"
- 若答得流畅 → "`defer func() { recover() }()` 能捕获 panic 吗？为什么？"

**追问参考答案：**
- defer 语句执行时参数就求值了，不是函数 return 时
- 能捕获；recover 在 defer 的直接调用里，panic 时 defer 执行，recover 生效

---

### 4. for range 与 goroutine

**面试题**：`for i, v := range slice` 里，在循环里写 `go func() { println(v) }()` 会有什么问题？怎么修复？

**参考答案要点：**
- **问题**：v 是循环复用的变量，闭包捕获的是引用，goroutine 执行时 v 可能已变成最后一个值，输出不确定
- **修复**：循环内 `v := v` 或 `val := v` 再在 goroutine 里用 val；或传参 `go func(x int) { println(x) }(v)`

**追问入口：**
- 若答出 → "用 `println(i)` 呢？i 有同样问题吗？"
- 若答得流畅 → "无限制起 goroutine 会有什么问题？"

**追问参考答案：**
- i 也是复用变量，闭包捕获同有问题，同样要 `i:=i` 或传参
- 调度开销大、内存涨、可能把机器打满；实际项目常用 worker pool 控制并发

---

### 5. Gin 中间件

**面试题**：Gin 的 `c.Next()` 和 `c.Abort()` 分别做什么？如果中间件里不调用 `c.Next()`，请求会怎样？

**参考答案要点：**
- **Next()**：调用下一个 handler，形成链式调用
- **Abort()**：终止后续 handler，不再调用 Next
- **不调用 Next**：后续 handler 不执行，请求提前结束（例如鉴权失败时）

**追问入口：**
- 若答出 → "你们项目里用了哪些中间件？鉴权、限流是怎么做的？"
- 若答得流畅 → "Gin 的 Context 是复用的，在中间件里存数据要注意什么？"

**追问参考答案：**
- 结合候选人项目（如 JWT 鉴权、限流中间件）
- 请求结束后 Context 会被回收，不要存指针引用；存值类型或拷贝，避免下次请求读到脏数据
