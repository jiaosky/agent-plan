你同事说的这几个点，方向是对的，但**不完整**。而且“SEO CHANNEL 通信”我怀疑他说的可能是 **SSE / Stream / Channel 通信**，不是 SEO。大模型后端里常说的是：**流式输出、事件通道、工具调用通道、MCP 通信、WebSocket/SSE、沙盒执行、文件上下文管理**。

你现在传统后端出身，最适合切入的不是“算法研究岗”，而是：

> **大模型应用后端 / Agent 平台后端 / RAG 工程 / AI Infra 工程 / 大模型产品工程**

这类岗位本质还是工程岗，但要求你懂 LLM 的交互范式、上下文、工具调用、检索、评测、成本、稳定性。

# 1️⃣ 你同事说的几个词，分别要学到什么程度

## MCP：必须懂

MCP 可以理解成：

> **大模型应用连接外部工具、数据源、系统能力的一套标准协议。**

它不是模型本身，也不是 Agent 框架，而是让模型可以访问外部系统的协议层。

你要掌握这些关键词：


| 概念         | 你要会说                                   |
| ---------- | -------------------------------------- |
| MCP Host   | AI 应用本体，比如 Claude Desktop、IDE、Agent 平台 |
| MCP Client | Host 里负责连接某个 MCP Server 的客户端           |
| MCP Server | 暴露工具、资源、Prompt 的服务                     |
| Tools      | 可执行动作，比如查库、调接口、读文件                     |
| Resources  | 上下文数据，比如文件、数据库记录、API 响应                |
| Prompts    | 可复用提示词模板                               |
| Transport  | stdio、本地进程通信、Streamable HTTP、远程通信      |
| JSON-RPC   | MCP 底层消息格式                             |
| tools/list | 工具发现                                   |
| tools/call | 工具执行                                   |


MCP 官方文档明确说明，MCP 是 client-server 架构，Host 会为每个 MCP Server 创建 MCP Client；MCP 数据层基于 JSON-RPC，核心 primitives 包括 tools、resources、prompts；传输层支持 stdio 和 Streamable HTTP。([modelcontextprotocol.io](https://modelcontextprotocol.io/docs/concepts/architecture))

你面试时可以这样说：

> MCP 解决的是“大模型应用如何标准化连接外部工具和上下文”的问题。传统后端里我可能直接写 Feign / HTTP Client 调业务接口，但在 Agent 场景下，需要把这些能力抽象成 tool/resource/prompt，并通过协议暴露给模型运行时。模型不是直接访问数据库，而是由宿主应用做工具发现、权限控制、参数校验、工具执行和结果回填。

这句话很加分。

---

## SSE / Channel 通信：必须懂

如果你同事说的是 “SEO”，大概率是口误，实际是 **SSE**。

大模型应用里，SSE 非常常见，因为模型输出不是一次性返回，而是 token-by-token 流式返回。

你要掌握：


| 点                   | 说明                       |
| ------------------- | ------------------------ |
| SSE                 | 服务端单向推送，适合 ChatGPT 式流式输出 |
| WebSocket           | 双向通信，适合实时协作、语音、复杂交互      |
| Chunk               | 模型分片输出                   |
| Delta               | 本次增量内容                   |
| Finish Reason       | stop、length、tool_calls 等 |
| Tool Call Streaming | 模型边生成边决定调用工具             |
| Backpressure        | 前端消费慢、服务端生成快时的处理         |
| Cancel              | 用户点停止生成，后端要中断请求          |
| Timeout             | 长连接超时、网关超时、客户端断开         |
| Trace ID            | 每次对话链路追踪                 |


传统后端里你可能习惯：

> 请求 → 处理 → 返回 JSON

大模型后端更多是：

> 请求 → 创建会话 → 调模型 → 流式输出 → 中途可能工具调用 → 工具结果回填 → 模型继续生成 → 结束事件

所以要能画出这种链路。

## 沙盒：必须懂，但不用一开始就实现很深

沙盒就是：

> **给 Agent 一个隔离的执行环境，让它可以读写文件、运行命令、执行代码、生成报告或改代码，但不能污染宿主系统。**

OpenAI 文档里对 Sandbox Agents 的定位很清楚：当 Agent 需要文件、命令、包、端口、快照、可恢复状态时使用 sandbox；如果只是短文本响应，就不需要 sandbox。([OpenAI开发者](https://developers.openai.com/api/docs/guides/agents))

官方也强调，沙盒适合处理目录级文档、生成文件、运行命令、生成 Markdown/CSV/JSONL/截图/网站预览，以及需要人工审核后继续恢复的工作流。([OpenAI开发者](https://developers.openai.com/api/docs/guides/agents/sandboxes))

你要会说这些设计点：


| 能力   | 工程关注点                         |
| ---- | ----------------------------- |
| 文件隔离 | 每个任务独立 workspace              |
| 命令执行 | 白名单、超时、资源限制                   |
| 网络访问 | 是否允许外网、是否允许访问内网               |
| 权限控制 | 不把密钥直接暴露给模型                   |
| 结果回收 | 产物文件、日志、执行结果如何返回              |
| 安全   | 防止 prompt injection、命令注入、数据泄漏 |
| 生命周期 | 创建、挂载、执行、快照、销毁                |


面试表达：

> 沙盒不是简单 exec shell，而是模型动作的安全边界。宿主服务负责鉴权、编排、状态和敏感资源；沙盒只拿到完成任务所需的最小文件和命令权限。

---

## 文件管理：必须懂

大模型公司的“文件管理”不是普通上传下载这么简单，而是围绕 **上下文工程**。

你要理解：


| 模块        | 重点                       |
| --------- | ------------------------ |
| 文件上传      | 大文件、断点续传、格式校验、病毒扫描       |
| 文件解析      | PDF、Word、Excel、图片、音视频转文本 |
| Chunking  | 文档切片策略                   |
| Embedding | 向量化                      |
| Index     | 向量索引、关键词索引、混合检索          |
| Retrieval | TopK、过滤条件、重排             |
| Citation  | 回答引用来源                   |
| 权限        | 用户只能检索自己有权限的文件           |
| 生命周期      | 文件删除后向量索引也要清理            |
| 多租户       | 企业知识库隔离                  |
| 增量更新      | 文件变更后的索引更新               |


OpenAI 的 File Search 文档也说明，文件搜索的核心是把外部知识接入模型，包括文档解析、切块、embedding、向量和关键词检索。([OpenAI开发者](https://developers.openai.com/api/docs/assistants/tools/file-search))

你需要能解释 RAG：

> RAG = Retrieval Augmented Generation，先从外部知识库检索相关上下文，再把检索结果拼进模型输入，让模型基于外部知识回答。

但是要进一步说：

> RAG 的难点不在“调 embedding 接口”，而在文件解析质量、切片策略、召回率、重排、权限过滤、引用溯源、幻觉控制、评测闭环和成本控制。

这才像工程师。

# 2️⃣ 同事说的是否有遗漏？有，而且不少

你要补齐这些核心知识。

## 大模型交互基础

你必须掌握：


| 知识                 | 要会到什么程度                                     |
| ------------------ | ------------------------------------------- |
| Prompt             | system / developer / user / tool message 区别 |
| Token              | 上下文长度、输入 token、输出 token、费用                  |
| Context Window     | 不是“记忆”，是一次请求可放入的上下文                         |
| Temperature        | 控制随机性                                       |
| Top-p              | 采样范围                                        |
| Max Tokens         | 输出上限                                        |
| Stop               | 终止符                                         |
| JSON Mode          | 只保证合法 JSON，不保证 schema                       |
| Structured Outputs | 按 JSON Schema 输出                            |
| Function Calling   | 模型决定调用哪个工具                                  |
| Tool Result        | 工具执行结果回填模型                                  |
| Multi-turn         | 多轮上下文压缩、摘要、裁剪                               |
| Streaming          | 流式输出                                        |
| Retry              | 模型调用失败、限流、超时重试                              |
| Rate Limit         | QPS、TPM、RPM                                 |
| Cost               | token 成本、缓存、批处理                             |
| Safety             | 内容安全、越权、防注入                                 |


Structured Outputs 和 JSON Mode 的区别要会说。OpenAI 文档明确指出，Structured Outputs 是 JSON mode 的演进；JSON mode 只保证输出是合法 JSON，而 Structured Outputs 还保证符合 schema。([OpenAI开发者](https://developers.openai.com/api/docs/guides/structured-outputs))

面试里很容易问：

> 为什么不用普通 JSON 输出？

你可以答：

> 普通 JSON 输出不稳定，可能字段缺失、类型错误、枚举越界。生产系统里如果下游依赖结构化结果，应该使用 JSON Schema / Structured Outputs，并在服务端做二次校验和降级处理。

## Agent 范式

现在大模型公司很常见的是 Agent 工程。

OpenAI 对 agent 的定义是：能够计划、调用工具、在多个 specialist 间协作，并保持足够状态完成多步骤工作的应用。([OpenAI开发者](https://developers.openai.com/api/docs/guides/agents))

你要理解几种典型 Agent 范式：


| 范式                         | 说明                                              |
| -------------------------- | ----------------------------------------------- |
| 单轮 LLM 调用                  | 最简单，输入 prompt，返回结果                              |
| RAG 问答                     | 检索知识库后回答                                        |
| Tool-using Agent           | 模型可调用外部工具                                       |
| Workflow Agent             | 明确流程编排，比如 classify → retrieve → answer → verify |
| Multi-agent                | 多个角色协作，比如 planner、executor、reviewer             |
| Human-in-the-loop          | 关键动作前需要人确认                                      |
| Code Agent                 | 可以读写代码、执行测试、修复问题                                |
| Browser/Computer-use Agent | 可以操作 UI、浏览网页、执行点击                               |
| Data Agent                 | 查库、生成 SQL、分析表格                                  |
| Long-running Agent         | 长任务、可暂停、可恢复                                     |


你现在后端背景，最好主攻：

> **RAG + Tool Calling + Agent Workflow + 沙盒文件处理**

这比直接卷模型训练更现实。

---

## 评测与可观测性：非常容易被忽视，但很重要

很多传统后端工程师转 AI，最容易显得“外行”的地方就是：只会说调模型，不会说怎么评测。

大模型系统不是确定性系统，所以必须有：


| 方向      | 指标                 |
| ------- | ------------------ |
| RAG 召回  | Recall@K、MRR、NDCG  |
| 回答质量    | 正确性、完整性、忠实性        |
| 幻觉率     | 是否编造               |
| 引用准确率   | 引用内容是否支撑答案         |
| 工具调用准确率 | 是否调用正确工具、参数是否正确    |
| 延迟      | 首 token 时间、总响应时间   |
| 成本      | 输入/输出 token、工具调用成本 |
| 稳定性     | 超时、重试、失败率          |
| 安全      | 越权、注入、敏感信息泄漏       |


LangChain 的 2026 Agent Engineering 调研提到，57.3% 受访组织已经有 Agent 在生产环境运行，质量是主要生产阻碍之一；同时 89% 的组织已实现某种 Agent 可观测性，62% 有详细 tracing。([LangChain](https://www.langchain.com/state-of-agent-engineering))

这说明：**生产级 AI 系统一定要讲 tracing、eval、灰度、回放、数据集。**

你面试可以说：

> 大模型应用不能只看接口成功率。要把每次请求拆成模型调用、检索、rerank、tool call、最终回答几个阶段做 tracing，同时沉淀 badcase 数据集，用离线 eval 和线上监控共同判断版本是否可上线。

这个很像“懂生产”。

# 1️⃣0️⃣ 面试里最容易被“嫌蠢”的点

直接说重点。

## 不要说：“大模型就是调接口”

应该说：

> 调模型只是最小闭环，生产系统还涉及上下文构造、工具调用、权限控制、流式通信、评测、观测、成本和安全。

---

## 不要把 Prompt 当成万能

应该说：

> Prompt 是一部分，更核心的是上下文工程，包括检索、压缩、结构化输入、工具结果组织和多轮状态管理。

---

## 不要说 RAG 就是向量数据库

应该说：

> 向量库只是 RAG 的一个组件。RAG 的质量取决于解析、切片、召回、重排、上下文拼接、引用和评测。

---

## 不要忽略权限

大模型系统很怕越权。

你要说：

> 模型不能绕过业务权限。工具调用、文件检索、知识库召回都必须经过用户、租户、角色和数据权限过滤。

---

## 不要忽略成本

你要会说：

> 大模型系统的成本主要来自输入 token、输出 token、embedding、rerank、工具调用和长上下文。需要通过缓存、摘要、裁剪、模型路由和批处理控制成本。

---

## 不要忽略不确定性

传统后端是确定性逻辑，大模型不是。

你要说：

> 大模型输出具有不确定性，所以生产环境需要结构化输出、schema 校验、guardrail、eval、badcase 回放和人工审核机制。
