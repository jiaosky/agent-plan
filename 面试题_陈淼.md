## 使用说明

- **考察重点**：近期对 Java/Go 工程师面试放宽要求，但对资深工程师/架构师，更关注项目经验和架构能力，主要考察：
  1. **能否清晰描述**最近几家公司的核心项目、负责的模块和日常产出
  2. **涉及的技术重点**：高并发架构、API 与系统对接、消息队列、K8s、nginx、Docker 容器、微服务治理等
  3. **架构与选型思路**：为什么这么设计、踩过哪些坑、如何权衡成本与复杂度
- **HC 工作内容**：承担主线产品在项目上的**定制、对接**和核心技术方案设计，需要有成熟的项目落地经验
- **期望互动**：引导候选人结合「最近 3 家公司/项目」展开，重点听其讲清业务背景、技术方案、问题与复盘
- **评分参考**：能讲清 1-2 个复杂项目的来龙去脉 + 做过关键决策/优化 + 沟通有条理 = 通过；只停留在概念、不清楚自己负责什么 = 需警惕

## 一、工作职责与项目经历

### 1. 工作职责描述

**面试题**：请简单介绍一下你目前/最近一份工作的主要职责，日常主要负责哪些模块？和产品、前端、运维是怎么协作的？

**考察要点：**

- 能否清晰说出自己的角色和产出
- 是否了解上下游协作流程

**参考答案：**

- 合格回答应能讲清岗位角色、负责模块、日常决策范围，而不是只说“负责后端开发”
- 会明确与产品、前端、测试、运维的协作边界，例如需求评审、接口定义、上线发布、故障处理
- 资深候选人通常会补充自己在方案评审、排期拆分、风险识别上的职责

**怎么追问：**

- 你说你负责核心模块，核心体现在哪？是流量、复杂度还是业务价值？
- 你和产品有分歧时怎么处理？举一个真实例子
- 你日常最有价值的产出是写代码、定方案，还是推动协作？为什么？

**面试题**：liveme 海外直播、RTA 广告系统、数字资产管理系统这些项目里，你具体负责哪些模块？作为架构师/高级工程师，你的主要产出是什么？

**考察要点：**

- 如果真是架构师或高级工程师，通常会提到方案设计、性能优化、故障治理、迁移改造或规范建设

**怎么追问：**

- 这几个项目里，哪个是你主导程度最高的？你拍过什么关键决策？

**面试题**：RTA 系统 10w QPS、单机 1.5w QPS，能简单说说整体架构和你的参与度吗？不用讲太细，说清主要设计思路即可。

**考察要点：**

**怎么追问：**

- 你负责的是全链路，还是其中某几层？边界是什么？

## 二、API 与系统对接（重点考察）

### 3. API 开发经验

**面试题**：你开发过哪些对外或对内的 API？RESTful 或 RPC 接口一般怎么设计？和前端、其他服务联调时，有没有遇到过参数不一致、跨域等问题？

**考察要点：**

- 有 API 开发经验
- 能说清常见问题及处理方式

**参考答案：**

- 应能说清接口设计原则：资源边界、幂等性、错误码、版本管理、鉴权、超时与重试
- 对联调问题，至少能提到参数契约不一致、枚举值漂移、时间格式、签名、跨域、兼容性
- 资深候选人会更关注接口治理，而不是只说“和前端对一下文档”

**怎么追问：**

- 你怎么保证接口定义在多人协作下不失控？
- 对外 API 升级时，怎么避免把老调用方弄挂？
- 如果前端说是后端问题，后端说是前端问题，你怎么快速定界？

---

**面试题**：liveme 直播平台、matchU 交友平台、数字资产管理系统，这些系统之间有没有对接？对接流程大概是怎样的？

**考察要点：**

- 有系统对接经历
- 能说清对接流程

**参考答案：**

- 好答案通常会按“需求确认 -> 接口契约 -> 联调 -> 验收 -> 上线 -> 监控”来讲
- 会说明同步调用、异步通知、回调签名、重试策略、幂等处理等关键点
- 真做过复杂对接的人会关注异常分支，而不是只说“双方按文档开发”

**怎么追问：**

- 对接中最难统一的通常是什么？字段、状态机，还是时序？
- 如果对方系统不稳定，你怎么设计重试和兜底？
- 你们是怎么做联调环境隔离和验收对账的？

---

### 4. 系统对接经验

**面试题**：RabbitMQ 替换 AWS SNS/SQS，迁移时有什么注意事项？和下游消费方的对接是怎么做的？

**考察要点：**

- 有对接/迁移经历
- 能说清基本流程和注意事项

**参考答案：**

- 至少应提到消息模型差异、消费确认、重试、死信、顺序性、幂等等核心点
- 会说明迁移不是一次切换，而是要有灰度、回滚、对账和验收
- 如果只回答“改代码切到 RabbitMQ”基本不够

**怎么追问：**

- 上下游消费语义不一致时，你们怎么抹平差异？
- 迁移期间有没有双写或双读？为什么这么做？
- 如果业务已经依赖旧队列的重试行为，切换后怎么避免语义变化？

---

**面试题**：数字资产管理系统里，业务层通过 RabbitMQ 与 Golang 后台服务通信，这种异步对接的优缺点是什么？和直接 HTTP 调用相比呢？

**考察要点：**

- 了解异步对接
- 能说清选型考虑

**参考答案：**

- 优点一般包括削峰、解耦、提升吞吐、隔离下游波动
- 缺点应能讲到链路变长、状态变复杂、排障困难、最终一致性和补偿成本
- 资深候选人会强调“是否适合异步”取决于业务时效和事务边界，不是默认更高级

**怎么追问：**

- 这个场景如果改成 HTTP 直调，会先死在哪？
- 异步链路出了问题，调用方怎么知道业务还没完成？
- 你们怎么做消息积压监控和失败补偿？

## 三、Nginx 与部署（重点考察）

### 5. Nginx 使用经验

**面试题**：Nginx 在项目里用来做什么？liveme 用 Kong 做 API 网关，Kong 和 Nginx 有什么关系？反向代理、负载均衡这些有接触过吗？

**考察要点：**

- 有 Nginx/网关使用经验
- 能说清基本用途

**参考答案：**

- 至少能说出反向代理、负载均衡、SSL 终止、静态资源、限流、统一入口等用途
- 如果提到 Kong，应知道 Kong 本质上是基于 Nginx/OpenResty 的网关能力扩展
- 真做过网关治理的人会区分接入层职责和业务层职责

**怎么追问：**

- 你们为什么要用 Kong，而不是直接维护 Nginx 配置？
- 哪些逻辑绝不能沉到网关里？
- 如果网关层 RT 抖动，你先排查哪些方向？

---

**面试题**：有没有配置过 Nginx 或 Kong 的 location、proxy_pass、限流等？跨域（CORS）怎么配置？

**考察要点：**

- 有配置经验更好
- 了解常见配置即可

**参考答案：**

- 至少应知道 location 匹配、proxy_pass 转发、header 透传、超时配置、基础限流思路
- 对 CORS，至少要讲到 origin、method、header、credentials、预检请求
- 真做过的人通常会提到线上变更风险，而不是只背配置项名字

**怎么追问：**

- `proxy_pass` 少了或多了 `/`，会有什么差异？
- CORS 为什么有时本地能调，线上不行？
- 限流如果配错了，最直接的业务后果是什么？

### 6. 部署与发布

**面试题**：liveme 基于 K8s 部署，你参与了多少？Docker 镜像构建、容器运行、多环境配置这些是怎么管理的？

**考察要点：**

- 了解 K8s/Docker 部署
- 能说清大致流程

**怎么追问：**

## 六、建议补充的深挖题（更适合 10 年以上候选人）

> 这一组题建议替换掉部分“介绍型”问题。目标不是确认他听过哪些技术，而是判断他是否真的做过高并发架构、复杂系统迁移和跨团队方案设计。

### 9. RTA 广告系统高并发深挖

**面试题**：你提到 RTA 系统峰值 10w QPS、单机 1.5w QPS。先别讲技术名词，按请求链路说一下：一个请求从接入到返回，中间经过哪些核心模块？最容易成为瓶颈的是哪一段，为什么？

**参考答案：**

- 正常回答应能讲清入口层、规则/画像查询、候选召回、过滤排序、缓存/存储、回包链路
- 会主动区分 CPU 型瓶颈、IO 型瓶颈、网络瓶颈、下游依赖瓶颈
- 会提到广告场景通常有严格 RT 要求，很多设计不是追求吞吐最大，而是追求尾延迟可控
- 如果真做过，通常会提到降级、超时控制、热点数据、本地缓存或异步更新，而不是只说“加机器、加 Redis”

**怎么追问：**

- 10w QPS 是怎么压出来的？压测流量模型和真实流量差异在哪？
- 单机 1.5w QPS 的前提条件是什么？CPU、核数、包体大小、命中率分别是多少？
- 如果 P99 抖动很大，但平均 RT 很漂亮，你更关注哪个指标？为什么？
- 假设下游画像服务 RT 从 5ms 升到 30ms，你怎么保护主链路？

---

### 10. 高并发容量设计与数据一致性

**面试题**：如果让你重新设计这个 RTA 系统，你会先做哪些容量假设？哪些数据必须强一致，哪些可以最终一致？

**参考答案：**

- 能先讲业务约束，再讲技术设计，比如 QPS、超时预算、可接受丢失比例、流量波峰波谷
- 能区分配置数据、画像数据、预算数据、投放状态、日志回传这几类数据的一致性要求
- 一般会把“实时决策链路”与“异步统计结算链路”分开设计
- 对资深候选人，理想回答会提到幂等、重试、副作用隔离、缓存与数据库双写风险

**怎么追问：**

- 哪一类数据你绝不能接受旧数据？为什么？
- 如果预算扣减用了缓存，怎么避免超发？
- 如果要在“低延迟”和“强一致”之间取舍，你怎么跟业务方解释？

---

### 11. RabbitMQ 替换 SNS/SQS 的迁移设计

**面试题**：RabbitMQ 替换 AWS SNS/SQS，这不是简单的中间件替换。你当时最担心的迁移风险是什么？怎么保证迁移期间业务不停机、不丢消息、不重复消费到出事故？

**参考答案：**

- 好答案会先讲语义差异：投递模型、消费确认、重试机制、死信、顺序性、可观测性
- 会提到灰度迁移、双写/双投递、影子消费、对账、回滚方案，而不是一次性切换
- 会主动说明消息幂等方案，例如业务主键、去重表、消费状态表
- 会提到“迁移完成”的验收标准，而不只是“功能能跑通”

**怎么追问：**

- 你们有没有做双写？如果双写，如何处理新老链路消费速度不同？
- 怎么证明没有丢消息？靠日志还是靠业务对账？
- 遇到重复消费时，业务侧怎么兜底？
- 如果 RabbitMQ 集群短暂不可用，生产侧和消费侧分别怎么处理？

### 12. 异步架构边界与失败处理

**面试题**：数字资产管理系统里，业务层通过 RabbitMQ 和 Golang 后台服务通信。哪些场景适合异步，哪些不适合？如果消息已经发出，但下游处理失败，怎么保证调用方感知和业务闭环？

**参考答案：**

- 适合同步改异步的通常是削峰、解耦、非强实时、允许状态中间态的场景
- 不适合的是用户强依赖即时结果、事务边界不能拆、失败需要立即回滚的场景
- 资深候选人会讲“业务状态机”，例如待处理、处理中、成功、失败、补偿中
- 理想回答会涉及回查、补偿、死信队列、告警、人工介入、幂等和可重放

**怎么追问：**

- 用户界面上怎么展示“请求已受理但未完成”？
- 补偿是自动补还是人工补？阈值怎么定？
- 如果消息消费成功了，但回写数据库失败，会出现什么问题？怎么补？

### 13. Redis Cluster 与热点问题

**面试题**：你简历里提到 Redis Cluster。你们在高并发项目里把 Redis 用在什么位置？如果出现热点 key、big key、缓存雪崩，你分别怎么判断和处理？

**参考答案：**

- 合格回答应能区分缓存、计数器、分布式锁、会话、排行榜等不同用途
- 对热点 key，应该能说出本地缓存、拆 key、随机过期、读写分离或限流保护
- 对 big key，应该能提到编码方式、拆分结构、扫描与删除成本、网络放大
- 对雪崩，应该能说预热、过期打散、多级缓存、降级兜底

**怎么追问：**

- Redis Cluster 下热点集中到单分片时，你怎么处理？
- 你怎么在线发现 big key？发现后怎么平滑治理？
- 如果缓存命中率下降，但数据库暂时没挂，你会先看哪些指标？

### 14. ES 使用是否停留在“会用”层面

**面试题**：你提到 ES。你们是把 ES 当搜索引擎、日志检索，还是分析引擎？索引设计时你最关注 mapping、分片数、刷新策略中的哪些点？

**参考答案：**

- 真做过的人会先说业务用途，再说写入模型和查询模型
- 会知道分片不是越多越好，mapping 一旦失控会影响存储和查询
- 如果是高写入场景，会提 refresh interval、批量写入、冷热分层、索引生命周期
- 如果是搜索场景，会提倒排、字段类型、keyword/text 区分、聚合代价

**怎么追问：**

- 分片数一开始定错了，后面怎么调整？
- ES 查询很慢时，你先看 DSL、mapping 还是集群资源？为什么？
- 为什么有些字段要存 keyword，有些要存 text？

### 15. Kong / Nginx 网关层的治理能力

**面试题**：liveme 用 Kong 做 API 网关。你如果是这个系统的负责人，网关层除了转发请求，还应该承担哪些治理职责？哪些能力不应该堆到网关里？

**参考答案：**

- 应该能说出鉴权、限流、熔断、灰度、观测、路由编排、统一日志等
- 更好的答案会强调边界：复杂业务规则不应堆在网关，避免网关膨胀成“巨石入口”
- 会提到插件化、配置变更风险、全局故障放大效应
- 资深候选人会关注网关性能成本和变更发布风险

**怎么追问：**

- 限流放在网关和放在应用侧，各自优缺点是什么？
- 网关配置发错导致全站故障，你会怎么设计回滚？
- 如果某个租户流量异常暴涨，如何只隔离它，不影响全局？

### 16. K8s 落地经验真伪判断

**面试题**：你提到基于 K8s 部署。你参与的是“把服务放上去”，还是参与了资源治理、发布策略、故障恢复？请讲一个你亲自处理过的线上案例。

**参考答案：**

- 如果只是使用者，通常只会说 Deployment、Service、镜像发布
- 真正做过治理的人会提到 requests/limits、HPA、探针、滚动发布、配置管理、Pod 驱逐、节点问题
- 更好的回答会把故障现象、根因、定位过程、改进动作讲完整
- 能说出“发布成功不代表服务可用”，说明有线上经验

**怎么追问：**

- readiness 和 liveness 分别配置错了会怎样？
- 为什么有些服务 CPU 不高，但还是频繁重启？
- 滚动发布过程中，如何避免新版本把下游打挂？

### 17. DDD 是否只是写在简历上

**面试题**：你简历写了 DDD。请不要先讲概念，直接说你在哪个项目里按什么边界拆了领域，聚合根是什么，拆分后解决了什么问题，又引入了什么成本？

**参考答案：**

- 真理解 DDD 的候选人会从业务边界、统一语言、聚合一致性边界切入
- 会承认 DDD 不是银弹，小团队或简单 CRUD 未必值得重建模型
- 好答案通常会提到领域服务、应用服务、防腐层、上下文边界，而不是只说“代码更优雅”
- 如果完全答不出聚合边界和交易边界，基本说明只停留在概念层

**怎么追问：**

- 你怎么判断一个对象该做实体还是值对象？
- 为什么这个动作要放在领域层，而不是 service 层？
- 如果两个聚合要跨边界协同，最终怎么落地？

### 18. 架构师角色的取舍与复盘

**面试题**：你做了多年架构和高级工程师。如果让你讲一个“当时方案看起来合理，后来发现有明显问题”的案例，你会讲哪个？问题出在架构、流程、沟通，还是预估失误？

**参考答案：**

- 这个问题主要看复盘能力，不是看他有没有失败
- 好答案应该能承认方案局限，并清楚说明当时的约束条件
- 会区分“技术判断错”还是“组织协作没跟上”，并给出后续修正动作
- 如果只讲成功、不讲代价，通常说明复盘深度不够

**怎么追问：**

- 如果今天重来一次，你会保留什么，推翻什么？
- 当时有没有人反对？你怎么做决策？
- 这个失误如果再次发生，你会提前看哪些预警信号？

## 七、建议优先使用的 8 题组合

如果面试时间只有 30-45 分钟，建议优先问下面 8 题，区分度最高：

1. RTA 请求链路和瓶颈定位
2. 10w QPS 的容量假设与压测方法
3. RabbitMQ 替换 SNS/SQS 的迁移风险与兜底
4. 异步架构的边界、补偿与状态闭环
5. Redis Cluster 热点与雪崩治理
6. Kong / Nginx 网关层该做什么、不该做什么
7. K8s 线上故障案例
8. DDD 在具体项目里的边界与代价

## 八、快速判断标准

- 如果回答一直停留在“用了什么组件”，没有容量、指标、边界、失败处理，深度不够
- 如果能持续讲清“为什么这样设计、成本是什么、失败怎么办、后来怎么改”，说明项目经验较实
- 对 10 年以上候选人，重点不是背定义，而是是否具备系统拆解、权衡取舍、故障复盘和跨团队推进能力

*建议：根据面试时间灵活选取 8-12 题，优先围绕最近 3 家公司的核心项目，重点问清「项目背景 → 负责模块 → 关键技术方案 → 关键指标 → 异常场景 → 复盘改进」。如果他能讲清“设计前提、指标目标、降级策略、迁移风险、复盘结论”，基本就比原题库更能判断真实深度。*
